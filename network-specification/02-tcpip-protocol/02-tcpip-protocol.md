# 1. TCP协议详解

## 1.1 TCP头部格式

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 头格式                                    │
└─────────────────────────────────────────────────────────────────┘

┌──────────┬──────────┬──────────┬───────────────────────────────┐
│  Source  │ Dest     │         Sequence Number                 │
│  Port    │ Port     │                                          │
│  16位    │ 16位     │         32位                            │
├──────────┼──────────┼──────────┼───────────────────────────────┤
│              Acknowledgment Number                             │
│                        32位                                    │
├──────────┼──────────┬──────────┬───────────────────────────────┤
│ Data     │ Flags    │      Window Size                       │
│ Offset   │ (9bits)  │           16位                          │
├──────────┼──────────┼──────────┼───────────────────────────────┤
│ Checksum │     Urgent Pointer                                 │
│  16位    │           16位                                      │
├──────────┼──────────┴──────────┴───────────────────────────────┤
│                    Options (可选, 最大40字节)                  │
├───────────────────────────────────────────────────────────────┤
│                           Data                                │
└───────────────────────────────────────────────────────────────┘

# 字段说明:
# Source Port: 源端口 (客户端随机选择, 服务端知名端口如80/443)
# Dest Port: 目标端口 (服务端监听端口)
# Sequence Number: 序列号 (字节偏移, 第一个字节的编号)
# Acknowledgment Number: 确认号 (期望收到的下一个字节)
# Data Offset: 数据偏移 (TCP头长度, 以4字节为单位)
# Flags: URG|ACK|PSH|RST|SYN|FIN
# Window Size: 窗口大小 (接收缓冲区可用空间)
# Checksum: 校验和 (头部+数据+伪头部)
# Urgent Pointer: 紧急指针 (URG标志置1时有效)
# Options: 选项 (MSS, Window Scale, SACK, Timestamp等)

# Flags:
# URG: 紧急指针有效
# ACK: 确认号有效
# PSH: 推送数据到应用层
# RST: 复位连接
# SYN: 同步序列号 (建立连接)
# FIN: 结束连接 (关闭连接)
```

```
# FIN_WAIT_2: 收到ACK, 等待对方FIN
# CLOSE_WAIT: 收到FIN, 已发送ACK, 等待本地关闭
# CLOSING: 双方同时关闭
# LAST_ACK: 最后一次ACK发送
# TIME_WAIT: 等待2MSL后关闭
```

# 2. TCP三次握手

## 2.1 握手过程

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 三次握手                                  │
└─────────────────────────────────────────────────────────────────┘

    客户端                              服务端
      │                                  │
      │         1. SYN (seq=x)          │
      │─────────────────────────────────►│ 客户端发送SYN, 请求建立连接
      │                                  │ seq=x表示已发送字节数
      │                                  │
      │         2. SYN+ACK              │
      │    (seq=y, ack=x+1)             │
      │◄─────────────────────────────────│ 服务端收到SYN, 发送SYN+ACK
      │                                  │ seq=y表示服务端字节流位置
      │                                  │ ack=x+1表示期望下次收到的字节
      │                                  │
      │         3. ACK                  │
      │    (seq=x+1, ack=y+1)          │
      │─────────────────────────────────►│ 客户端发送ACK确认
      │                                  │
      │     ESTABLISHED                  │ 双方都进入连接状态
      │◄────────────────────────────────►│ 可以开始传输数据
      │                                  │

# 为什么需要三次握手?
# 1. 确认双方的发送和接收能力正常
#    - 第一次: 客户端能发送, 服务端能接收
#    - 第二次: 服务端能发送, 客户端能接收
#    - 第三次: 双方都知道对方能发送也能接收
#
# 2. 同步双方的序列号
#    - 客户端生成初始序列号 x
#    - 服务端生成初始序列号 y
#    - 双方通过握手确认对方的序列号
#
# 3. 防止历史连接初始化
#    - 如果第一次SYN延迟到达, 服务端会发送SYN+ACK
#    - 客户端发现ack不符合, 发送RST
```

## 2.2 实战抓包分析

```bash
# 抓取TCP三次握手
sudo tcpdump -i eth0 'tcp[tcpflags] == tcp-syn' -n

# 完整抓包分析
sudo tcpdump -i eth0 'tcp' -n -A

# 查看握手过程 (过滤SYN, SYN+ACK, ACK)
sudo tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0' -n

# telnet测试握手
telnet example.com 80
# 按Ctrl+]退出

# curl测试
curl -v http://example.com
# 可以看到 TCP connection established -> HTTP request/response
```

# 3. TCP四次挥手

## 3.1 挥手过程

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 四次挥手                                  │
└─────────────────────────────────────────────────────────────────┘

    客户端                              服务端
      │                                  │
      │         1. FIN (seq=u)           │
      │─────────────────────────────────►│ 客户端请求关闭连接
      │                                  │ FIN_WAIT_1
      │                                  │
      │         2. ACK (ack=u+1)        │
      │◄─────────────────────────────────│ 服务端确认, 等待数据发送完毕
      │                                  │ CLOSE_WAIT
      │                                  │
      │         (处理剩余数据)            │
      │                                  │
      │         3. FIN (seq=v)           │
      │◄─────────────────────────────────│ 服务端发送FIN
      │                                  │ LAST_ACK
      │                                  │
      │         4. ACK (ack=v+1)        │
      │─────────────────────────────────►│ 客户端确认
      │                                  │
      │         TIME_WAIT                │
      │    (等待2MSL=60秒)               │
      │                                  │ CLOSED
      │                                  │
      │         CLOSED                   │
      ▼                                  ▼

# 为什么需要四次挥手?
# 1. TCP是全双工通信
#    - 每个方向都需要单独关闭
#    - 客户端发送FIN表示"我不再发送数据了"
#    - 服务端发送FIN表示"我不再发送数据了"
#
# 2. TIME_WAIT作用 (2MSL = 60秒)
#    - 保证最后的ACK能到达对端
#    - 让旧连接的报文在网络中消失
#    - 避免影响新连接
```

## 3.2 挥手状态详解

```bash
# 查看连接状态
ss -tan state fin-wait-1
ss -tan state fin-wait-2
ss -tan state time-wait

# 查看TCP详细信息
cat /proc/net/tcp

# 查看各状态的连接数
ss -tan | awk '{print $1}' | sort | uniq -c

# 常见状态:
# ESTABISHED: 正常数据传输
# SYN_SENT: 正在建立连接
# SYN_RECV: 收到连接请求
# FIN_WAIT_1: 主动关闭
# FIN_WAIT_2: 等待对方关闭
# TIME_WAIT: 等待2MSL
# CLOSE_WAIT: 被动关闭等待
# LAST_ACK: 最后确认
# CLOSED: 已关闭
```

# 4. TCP可靠传输

## 4.1 确认应答机制

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 确认应答机制                              │
└─────────────────────────────────────────────────────────────────┘

发送方                              接收方
  │                                   │
  │─────── Data: seq=1000, len=100 ──►│ 收到1000-1100字节
  │                                   │
  │◄──────── ACK: ack=1100 ──────────│ 期望下一个字节1100
  │                                   │
  │─────── Data: seq=1100, len=100 ──►│ 收到1100-1200字节
  │                                   │
  │◄──────── ACK: ack=1200 ──────────│
  │                                   │
  │         (丢包情况)                │
  │─────── Data: seq=1300, len=100 ──►│
  │                                   │
  │◄──────── ACK: ack=1200 ──────────│ 确认号不变, 提醒重传
  │                                   │
  │─────── Data: seq=1200, len=100 ──►│ 收到1200-1300字节
  │                                   │
  │◄──────── ACK: ack=1400 ──────────│
  │                                   │

# 超时重传:
# 发送数据后启动定时器
# 超时未收到ACK, 重传数据
# 指数退避: 1s → 2s → 4s → 8s...
```

## 4.2 序列号机制

```bash
# 查看连接序列号信息
ss -i

# TCP序列号计算
# 初始序列号 (ISN): 每次连接时随机生成
# 当前序列号 = ISN + 已发送字节数

# 序列号回绕处理
# TCP使用相对序列号, 便于分析
tcpdump -N 'tcp' -v

# 查看TCP重传统计
netstat -s | grep -i retransmit
cat /proc/net/netstat | grep -i retransmit
```

# 5. 滑动窗口与流量控制

## 5.1 滑动窗口原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 滑动窗口                                 │
└─────────────────────────────────────────────────────────────────┘

发送方窗口状态:
┌─────────────────────────────────────────────────────────────────┐
│ [已发送并确认] │ [已发送未确认] │    [可用窗口]    │ [不允许发送] │
│   0-1000      │   1000-1200   │    1200-1600     │  1600+      │
└─────────────────────────────────────────────────────────────────┘
                  ▲              ▲
                  │              │
              发送指针         窗口左沿

# 窗口大小由接收方的Window Size决定
# Window Size = 接收缓冲区大小 - 未读数据

# 流量控制:
# 接收方告诉发送方: "我还有多少空间"
# 发送方不能超过这个限制发送数据

# 零窗口:
# 当接收方缓冲区满, Window Size = 0
# 发送方停止发送, 启动探测定时器
# 接收方窗口恢复后, 发送Window Update
```

## 5.2 窗口探测

```bash
# 查看TCP窗口
ss -tm

# 查看实际窗口大小
cat /proc/sys/net/ipv4/tcp_rmem   # 接收窗口
cat /proc/sys/net/ipv4/tcp_wmem   # 发送窗口

# 查看窗口探测
cat /proc/sys/net/ipv4/tcp_probe_threshold

# 调整窗口大小
# 临时调整
sysctl -w net.ipv4.tcp_rmem="4096 87380 6291456"

# 永久调整
echo "net.ipv4.tcp_rmem = 4096 87380 6291456" >> /etc/sysctl.conf
```

# 6. TCP拥塞控制

## 6.1 拥塞窗口

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 拥塞控制算法                              │
└─────────────────────────────────────────────────────────────────┘

拥塞窗口 (cwnd) vs 接收窗口 (rwnd):
- cwnd: 发送方自己维护, 表示能发送多少数据
- rwnd: 接收方告知, 表示能接收多少数据
- 实际发送窗口 = min(cwnd, rwnd)

四个算法:
1. 慢启动 (Slow Start)
   - cwnd从1开始, 指数增长
   - cwnd *= 2 (每收到一个ACK)
   - 直到达到慢启动阈值 (ssthresh)

2. 拥塞避免 (Congestion Avoidance)
   - cwnd线性增长
   - cwnd += 1 (每RTT)
   - 直到发生拥塞

3. 快速重传 (Fast Retransmit)
   - 收到3个重复ACK, 说明丢包
   - 不等超时, 立即重传

4. 快速恢复 (Fast Recovery)
   - 收到3个重复ACK后
   - ssthresh = cwnd / 2
   - cwnd = ssthresh + 3
   - 然后线性增长

# 图示:
# cwnd
#  ^
# 32│                        ●──●──●
# 16│              ●──●──●──●
#  8│     ●──●──●──●
#  4│●──●
#  2│●
#  1│●
#  0└──────────────────────────────► RTT
#      1   2   3   4   5   6   7   8
```

## 6.2 拥塞控制命令

```bash
# 查看当前cwnd
ss -i

# 查看网络统计
netstat -s | grep -i congestion

# 查看TCP参数
sysctl net.ipv4.tcp_congestion_control   # 当前算法
sysctl net.ipv4.tcp_slow_start_after_idle  # 空闲后是否重置

# 可用算法
sysctl net.ipv4.tcp_available_congestion_control

# 常见算法:
# cubic (Linux默认)
# reno (经典)
# bbr (Google开发)
#vegas, westwood, etc.

# 切换算法
sysctl -w net.ipv4.tcp_congestion_control=bbr

# 永久配置
echo "net.ipv4.tcp_congestion_control = bbr" >> /etc/sysctl.conf
```

# 7. UDP协议

## 7.1 UDP头部格式

```
┌─────────────────────────────────────────────────────────────────┐
│                    UDP 头格式                                    │
└─────────────────────────────────────────────────────────────────┘

┌──────────┬──────────┬──────────┬───────────────┐
│  Source  │ Dest     │  Length  │   Checksum    │
│  Port    │ Port     │          │               │
│  16位    │ 16位     │  16位    │    16位       │
└──────────┴──────────┴──────────┴───────────────┘

# UDP特点:
# 1. 无连接: 不需要握手/挥手
# 2. 不可靠: 不保证交付, 不排序
# 3. 简单高效: 头部仅8字节
# 4. 无流量/拥塞控制
```

## 7.2 TCP vs UDP

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP vs UDP                                    │
└─────────────────────────────────────────────────────────────────┘

┌────────────────┬────────────────┬────────────────────────────┐
│    特性         │      TCP        │          UDP               │
├────────────────┼────────────────┼────────────────────────────┤
│ 连接性         │ 面向连接        │ 无连接                      │
│ 可靠性         │ 可靠传输        │ 不可靠                      │
│ 顺序性         │ 保证顺序        │ 不保证顺序                  │
│ 速度           │ 较慢            │ 较快                        │
│ 头部大小       │ 20-60字节       │ 8字节                       │
│ 流量控制       │ 有              │ 无                          │
│ 拥塞控制       │ 有              │ 无                          │
│ 使用场景       │ HTTP, SSH, FTP │ DNS, VoIP, 视频, 游戏      │
│ Socket类型     │ SOCK_STREAM    │ SOCK_DGRAM                  │
└────────────────┴────────────────┴────────────────────────────┘

# 选择TCP还是UDP?
# 用TCP: 需要可靠传输, 数据完整, 顺序正确
# 用UDP: 实时性要求高, 能容忍少量丢包
#         (视频, 语音, 游戏, DNS, DHCP)
```

## 本章小结

- **TCP三次握手**: SYN → SYN+ACK → ACK, 确认双方能力, 同步序列号
- **TCP四次挥手**: FIN → ACK → FIN → ACK, 全双工关闭
- **可靠传输**: 序列号 + 确认应答 + 超时重传
- **流量控制**: 滑动窗口, 接收方控制发送方速度
- **拥塞控制**: 慢启动 → 拥塞避免 → 快速重传 → 快速恢复
- **UDP**: 无连接, 不可靠, 但高效, 用于实时场景

**关键命令回顾:**

```bash
# 查看连接
ss -tan, ss -ti, ss -tm

# 查看状态
netstat -tan, netstat -s

# 抓包分析
tcpdump 'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0'

# TCP参数
sysctl -a | grep tcp
cat /proc/net/tcp
```
