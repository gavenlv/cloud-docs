# GCP专家之路：深入原理与架构本质

## 本章导学

**这不是一本入门手册。**

市面上99%的GCP教程都在告诉你"怎么用"——点哪个按钮、输什么命令。但如果你不知道"为什么"，你永远只能照猫画虎，遇到实际问题就束手无策。

**学完本章后，你将能够：**

- 从**电气层面**理解云计算为什么比传统数据中心高效
- 从**协议层面**理解HTTPS/TLS的工作原理，不再对SSL证书心存疑惑
- 从**内核层面**理解虚拟机如何实现资源隔离
- 从**架构层面**理解VPC网络的设计哲学
- 从**分布式系统层面**理解IAM权限模型的评估逻辑
- 从**经济学层面**理解云成本的构成和优化本质

**学习方法：**

每一节都会按照这个结构展开：
```
原理 → 架构 → 协议细节 → 实际计算/演示 → 常见误区
```

让我们开始。

## 1.1 为什么云计算便宜？——从数据中心的电费说起

当你听到"云计算比自建数据中心便宜"时，你可能认为这是云厂商的营销话术。但让我们做一道数学题。

### 1.2.2 虚拟化的底层实现原理

**x86架构的虚拟化挑战**

```
x86虚拟化难题（1960年代-2000年代）：

┌─────────────────────────────────────────────────────────────────┐
│  问题：x86架构设计时没有考虑虚拟化                                │
│                                                                  │
│  特权级别机制：                                                  │
│  Ring 0：内核态（最高权限）- 操作系统运行在这里                   │
│  Ring 1：设备驱动                                                │
│  Ring 2：设备驱动                                                │
│  Ring 3：用户态（最低权限）- 应用程序运行在这里                  │
│                                                                  │
│  虚拟化要求：                                                    │
│  ├── VMM（虚拟机监控器）需要运行在Ring 0                        │
│  ├── 虚拟机OS需要运行在Ring 1（但Guest OS以为自己运行Ring 0）  │
│  └── 这在x86架构上无法实现                                       │
│                                                                  │
│  两种解决方案：                                                  │
│  1. 全虚拟化（Binary Translation）：                            │
│     ├── VMM动态翻译虚拟机发出的特权指令                         │
│     ├── 性能损失约20-30%                                        │
│     └── 早期VMware使用这种方式                                   │
│                                                                  │
│  2. 半虚拟化（Paravirtualization）：                            │
│     ├── 修改Guest OS，移除不可虚拟化的指令                       │
│     ├── 性能接近原生                                            │
│     └── Xen使用这种方式（早期AWS）                               │
│                                                                  │
│  3. 硬件辅助虚拟化（2005年后）：                                 │
│     ├── Intel VT-x / AMD-V 提供新特权级别                       │
│     ├── VMX root mode / VMX non-root mode                      │
│     ├── Guest OS无需修改                                        │
│     └── 现代虚拟化标配                                           │
└─────────────────────────────────────────────────────────────────┘
```

**现代虚拟化架构（KVM为例）**

```

│              完整虚拟化技术栈（以KVM为例）                        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                      用户空间（User Space）                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐            │
│  │  VM 1       │  │  VM 2       │  │  VM N       │            │
│  │  App        │  │  App        │  │  App        │            │
│  │  Guest OS   │  │  Guest OS   │  │  Guest OS   │            │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘            │
│         │                │                │                     │
│         └────────────────┼────────────────┘                     │
│                          │                                      │
│              ┌───────────┴───────────┐                         │
│              │    Hypervisor (KVM)   │                         │
│              │    - CPU虚拟化 (VT-x)  │                         │
│              │    - 内存虚拟化 (EPT)  │                         │
│              │    - I/O虚拟化 (VT-d) │                         │
│              └───────────┬───────────┘                         │
│                          │                                      │
│              ┌───────────┴───────────┐                         │
│              │    Linux Kernel      │                         │
│              │    - 调度器           │                         │
│              │    - 内存管理        │                         │
│              │    - 设备驱动        │                         │
│              └───────────┬───────────┘                         │
│                          │                                      │
│              ┌───────────┴───────────┐                         │
│              │    Hardware (x86)    │                         │
│              │    - CPU             │                         │
│              │    - Memory          │                         │
│              │    - NIC            │                         │
│              │    - Storage        │                         │
│              └───────────────────────┘                         │
└─────────────────────────────────────────────────────────────────┘

关键虚拟化技术：

1. CPU虚拟化（Intel VT-x）
   ├── VMX操作模式切换
   ├── VMCS（Virtual Machine Control Structure）保存状态
   └── 虚拟机以为自己运行在Ring 0，实际被硬件截获

2. 内存虚拟化（Extended Page Table, EPT）
   ├── Guest虚拟地址 → Guest物理地址 → Host物理地址
   ├── 两级页表，硬件自动转换
   └── 嵌套页表（Nested EPT）用于二级虚拟化

3. I/O虚拟化
   ├── VT-d：直接内存访问（DMA）重映射
   ├── 设备透传（PCI Passthrough）
   ├── 虚拟网卡（virtio）
   └── SR-IOV：单根I/O虚拟化（硬件支持）

### 1.2.2 深入理解CPU虚拟化：VMX模式切换

```
┌─────────────────────────────────────────────────────────────────┐
│              Intel VT-x：硬件辅助虚拟化                         │
└─────────────────────────────────────────────────────────────────┘

问题：x86架构有4个特权级别（Ring 0-3），但Guest OS也需要Ring 0。
      Hypervisor也需要Ring 0。一山不容二虎！

解决方案：引入VMX模式

┌─────────────────────────────────────────────────────────────────┐
│  两种新模式（VT-x引入）：                                        │
│                                                                  │
│  VMX root mode：                                                │
│  ├── Hypervisor运行在这里                                        │
│  ├── 拥有最高特权                                                │
│  └── 所有特权指令都可以执行                                      │
│                                                                  │
│  VMX non-root mode：                                            │
│  ├── Guest VM运行在这里                                          │
│  ├── 也有Ring 0-3                                               │
│  └── 执行特权指令时会被" VM Exit"截获                           │
└─────────────────────────────────────────────────────────────────┘

VMX操作流程：

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Guest VM（non-root）执行特权指令：                              │
│                                                                  │
│  1. Guest OS在VM内执行特权指令（比如CR3修改）                   │
│                                                                  │
│  2. CPU检测到这是敏感操作，自动触发VM Exit                      │
│                                                                  │
│  3. CPU切换到VMX root mode，跳转到Hypervisor                   │
│                                                                  │
│  4. Hypervisor处理这个操作（模拟或直接执行）                    │
│                                                                  │
│  5. Hypervisor执行VM Entry，返回Guest VM                        │
│                                                                  │
│  6. Guest VM继续执行，仿佛什么都没发生                          │
│                                                                  │
│  性能开销：每次VM Exit约1000-5000个CPU周期                      │
│  优化方向：减少VM Exit次数（比如批处理I/O）                     │
└─────────────────────────────────────────────────────────────────┘

VMCS（Virtual Machine Control Structure）：

┌─────────────────────────────────────────────────────────────────┐
│  VMCS是内存中的一块数据结构，保存VM状态：                        │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  VMCS组成部分：                                          │   │
│  │                                                         │   │
│  │  Guest State Area：                                      │   │
│  │  ├── Guest寄存器（RAX, RBX, RSP...）                   │   │
│  │  ├── Guest CR3（页表基址）                              │   │
│  │  ├── Guest RIP（指令指针）                             │   │
│  │  └── Guest RFLAGS                                       │   │
│  │                                                         │   │
│  │  Host State Area：                                       │   │
│  │  ├── Hypervisor寄存器（恢复时用）                       │   │
│  │  └── CR3、RIP等                                         │   │
│  │                                                         │   │
│  │  VMX Controls：                                         │   │
│  │  ├── 哪些事件触发VM Exit                               │   │
│  │  ├── Pin-Based VM-Execution Controls                  │   │
│  │  └── CPU-Based VM-Execution Controls                  │   │
│  │                                                         │   │
│  │  VM Exit Info：                                          │   │
│  │  ├── 记录VM Exit的原因                                 │   │
│  │  └── 退出时引发的异常信息                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  每个vCPU对应一个VMCS实例                                        │
│  VM正常运行时，VMCS在内存中；VM Exit时，CPU读写VMCS              │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────┐
│              EPT（Extended Page Table）内存虚拟化深度剖析         │
└─────────────────────────────────────────────────────────────────┘

为什么需要内存虚拟化？

普通内存访问（无虚拟化）：
┌─────────────────────────────────────────────────────────────────┐
│  虚拟地址 → 页表翻译 → 物理地址                                  │
│                                                                  │
│  CR3寄存器保存根页表地址                                        │
│  MMU硬件自动完成翻译                                            │
└─────────────────────────────────────────────────────────────────┘

虚拟化内存访问（两级翻译）：
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Guest VM内：                                                    │
│  ├── Guest Virtual Address (GVA) → Guest Physical Address (GPA)│
│  └── 由Guest OS的页表完成                                        │
│                                                                  │
│  Hypervisor管理：                                                │
│  └── GPA → Host Physical Address (HPA)                         │
│      由EPT页表完成（硬件自动翻译）                               │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  地址翻译流程                            │   │
│  │                                                         │   │
│  │   GVA ──→ GPA ──→ HPA                                   │   │
│  │    │       │       │                                     │   │
│  │    │       │       └── 最终的物理内存地址                 │   │
│  │    │       │             （真实硬件地址）                 │   │
│  │    │       └── Guest OS认为的"物理地址"                  │   │
│  │    │             （实际是虚拟化的）                      │   │
│  │    └── 应用程序看到的地址                               │   │
│  │                                                         │   │
│  │   Guest OS维护自己的页表（GVA→GPA）                      │   │
│  │   Hypervisor维护EPT页表（GPA→HPA）                      │   │
│  │   CPU硬件自动完成两级翻译，性能接近原生！                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

EPT页表结构（4级）：

┌─────────────────────────────────────────────────────────────────┐
│  EPTP（EPT Pointer）指向根页表                                  │
│                                                                  │
│  EPTP ──→ EPT Page Directory Pointer (PML3)                    │
│       ──→ EPT Page Directory (PML2)                            │
│       ──→ EPT Page Table (PML1)                                │
│       ──→ 4KB物理页面                                          │
│                                                                  │
│  每个条目64位，包含：                                           │
│  ├── 物理页框地址（4KB对齐）                                    │
│  ├── Read/Write/Execute权限                                     │
│  ├── Dirty/Accessed标志                                          │
│  └── EPT M模式（内存类型）                                     │
│                                                                  │
│  关键优势：                                                      │
│  ├── 内存翻译完全由硬件完成                                     │
│  ├── 无需软件模拟                                               │
│  ├── Hypervisor可以监控每个VM的内存访问                        │
│  └── 支持内存去重（KSM - Kernel Samepage Merging）             │
└─────────────────────────────────────────────────────────────────┘

内存虚拟化的三个阶段：

┌─────────────────────────────────────────────────────────────────┐
│  阶段1：软件模拟（1980s-2000s，早期虚拟化）                      │
│                                                                  │
│  问题：CPU不支持硬件虚拟化                                       │
│  解决：Hypervisor模拟内存访问                                   │
│                                                                  │
│  ├── Guest访问CR3 → VM Exit → Hypervisor模拟                  │
│  ├── 性能极差（每次内存访问都要退出）                          │
│  └── 全虚拟化时代的无奈选择                                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  阶段2：影子页表（2005-2008，过渡方案）                          │
│                                                                  │
│  思想：Hypervisor维护一个"影子页表"                              │
│  ├── 映射：GVA → HPA（直接）                                    │
│  ├── Guest页表变更时，Hypervisor同步更新影子页表                │
│  └── 内存访问不再VM Exit                                        │
│                                                                  │
│  问题：                                                          │
│  ├── 实现极其复杂（需要跟踪Guest所有页表操作）                  │
│  ├── 需要同步Guest和Host的页表                                  │
│  └── 实现bug导致稳定性问题                                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  阶段3：EPT/NPT（2008+，硬件支持）                               │
│                                                                  │
│  思想：CPU原生支持两级内存翻译                                   │
│                                                                  │
│  ├── 第一级：Guest页表（GVA → GPA）                            │
│  ├── 第二级：EPT页表（GPA → HPA）                              │
│  └── CPU硬件自动完成，无需软件介入                              │
│                                                                  │
│  优势：                                                          │
│  ├── 性能接近原生（overhead < 5%）                              │
│  ├── 实现简单（硬件处理所有细节）                               │
│  ├── Hypervisor可以设置内存权限兜底                            │
│  └── 成为现代虚拟化的标准                                       │
│                                                                  │
│  Google使用：                                                    │
│  └── 所有GCP VM都运行在KVM + EPT上                              │
│      这就是GCP VM性能接近物理机的原因！                         │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2.3 虚拟机 vs 容器：本质区别

这是最容易被误解的概念。让我从底层原理解释：

```
┌─────────────────────────────────────────────────────────────────┐
│                    进程隔离层级对比                               │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  虚拟机（VM）：                                                  │
│  ├── 完整隔离：每个VM有自己的内核                                │

│  │  Lib 1  │  │  Lib 2  │  │  Lib 3  │                         │
│  │ Guest OS│  │ Guest OS│  │ Guest OS│   ← 各自独立内核          │
│  └────┬────┘  └────┬────┘  └────┬────┘                         │
│       │            │            │                                │

│                    │                                             │
│              ┌─────┴─────┐                                       │
│              │ Hypervisor │                                       │
│              └─────┬─────┘                                       │
│                    │                                             │
│              ┌─────┴─────┐                                       │
│              │  Host OS   │                                       │
│              │  (Linux)   │                                       │
│              └─────┬─────┘                                       │

│              └───────────┘                                       │

┌─────────────────────────────────────────────────────────────────┐

│  容器（Container）：                                             │
│  ├── 进程隔离：所有容器共享宿主机内核                            │
│  ├── 启动时间：毫秒级                                            │
│  ├── 资源开销：约1%（namespaces + cgroups）                     │
│  ├── 启动介质：应用 + 依赖（MB级别）                             │
│  └── 隔离级别：内核级（namespace）                               │
│                                                                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│  │  App 1  │  │  App 2  │  │  App 3  │   ← 各自独立命名空间     │
│  │  Lib 1  │  │  Lib 2  │  │  Lib 3  │                         │
│  └────┬────┘  └────┬────┘  └────┬────┘                         │

│                   │                                              │
│           ┌──────┴──────┐                                        │

│           │   (Linux)   │   ← 共享内核                          │
│           └──────┬──────┘                                        │

│           ┌─────┴─────┐                                          │
│           │ Hardware  │                                          │
│           └───────────┘                                          │
└─────────────────────────────────────────────────────────────────┘

关键区别解释：

1. 为什么容器启动快？
   └── 因为不需要启动整个操作系统，只需启动进程

2. 为什么容器轻量？
   └── 因为共享内核和系统库，不需要重复加载

3. 为什么容器不安全？
   └── 因为共享内核，一个容器逃逸可能影响宿主机
       （VM有硬件级隔离，更安全）

4. 为什么K8s用容器而不是VM？
   └── 因为K8s设计用于微服务，需要快速启停
       毫秒级启动 vs 分钟级启动，差距巨大

### 1.2.4 容器底层原理：namespace、cgroups与文件系统

**理解容器，必须理解Linux内核的这三个机制。**

#### namespace：进程的"障眼法"

```
┌─────────────────────────────────────────────────────────────────┐
│              Linux namespace：让进程"看"不到其他进程             │
└─────────────────────────────────────────────────────────────────┘

namespace是Linux内核的一项功能，它可以"隔离"系统中某些资源的"视图"，
让处于某个namespace中的进程以为自己独占了这些资源。

┌─────────────────────────────────────────────────────────────────┐
│  六种namespace类型（Linux 5.6+）：                               │
│                                                                  │
│  ┌─────────────┬───────────────────────────────────────────┐   │
│  │  类型        │  隔离内容                                   │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  PID        │  进程ID空间                                 │   │
│  │  （容器内进程看不到宿主机进程）                          │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  Network    │  网络设备、端口、路由表                      │   │
│  │  （容器内有独立的网络协议栈）                            │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  Mount      │  文件系统挂载点                             │   │
│  │  （容器内有独立的根文件系统）                            │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  UTS        │  主机名、域名                               │   │
│  │  （容器可以有自己的hostname）                            │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  IPC        │  共享内存、信号量、消息队列                  │   │
│  │  （容器内进程通信与宿主机隔离）                          │   │
│  ├─────────────┼───────────────────────────────────────────┤   │
│  │  User       │  用户ID、组ID空间                          │   │
│  │  （容器内root映射到宿主机普通用户）                      │   │
│  └─────────────┴───────────────────────────────────────────┘   │
│                                                                  │
│  关键洞察：namespace不隐藏资源，只改变资源的"视图"               │
│  一个进程在namespace A中看到的是A视图，在namespace B中看到B视图 │
└─────────────────────────────────────────────────────────────────┘
```

**让我们通过实际操作理解namespace：**

```
┌─────────────────────────────────────────────────────────────────┐
│              namespace操作演示                                   │
└─────────────────────────────────────────────────────────────────┘

查看当前进程的namespace：

$ ls -la /proc/self/ns/
total 0
drwxr-xr-x 5 root root 4096 Mar 19 10:00 .
drwxr-xr-x 3 2 root root 4096 Mar 19 10:00 ..
lrwxrwxrwx 1 root root 0 Mar 19 10:00 cgroup -> cgroup:[4026531835]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 ipc -> ipc:[4026531839]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 mnt -> mnt:[4026531840]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 net -> net:[4026531841]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 pid -> pid:[4026531836]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 pid_for_children -> pid:[4026531836]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 user -> user:[4026531837]
lrwxrwxrwx 1 root root 0 Mar 19 10:00 uts -> uts:[4026531838]

每个数字inode代表一个namespace实例。
两个进程如果指向相同的inode，说明在同一个namespace中。

创建新的PID namespace（unshare）：

$ unshare --pid --fork --mount-proc /bin/bash
$ ps -ef
UID        PID  PPID  C STIME TTY          TIME CMD
root         1    0  0 10:00 pts/0    00:00:00 /bin/bash
root         2    1  0 10:00 pts/0    00:00:00 ps -ef

关键观察：
├── 现在ps显示的PID从1开始（而不是宿主机的PID）
├── 这是因为我们创建了新的PID namespace
├── 容器中的init进程（PID 1）是所有其他进程的祖先
└── 容器退出后，这个namespace中的所有进程都会被销毁
```

#### cgroups：资源的"紧箍咒"

```
┌─────────────────────────────────────────────────────────────────┐
│              cgroups：限制容器能用的资源上限                     │
└─────────────────────────────────────────────────────────────────┘

namespace解决了"看"的问题，但没解决"用多少"的问题。
cgroups（control groups）就是用来限制进程使用多少资源的机制。

┌─────────────────────────────────────────────────────────────────┐
│  cgroups v2层级结构：                                           │
│                                                                  │
│                    /                                           │
│                    ├── user.slice                              │
│                    │     ├── user-1000.slice                   │
│                    │     │     ├──app.slice                    │
│                    │     │     │   ├── app-1.service           │
│                    │     │     │   └── app-2.service           │
│                    │     └── session-1.scope                   │
│                    └── system.slice                            │
│                          ├── docker.service                    │
│                          └── kubelet.service                    │
│                                                                  │
│  特点：                                                          │
│  ├── 单根层级（cgroups v2，所有controller挂载到同一棵树）       │
│  ├── 子进程自动继承父节点的cgroups设置                          │
│  └── 资源限制是"上限"，不是"固定分配"                          │
└─────────────────────────────────────────────────────────────────┘

主要cgroups控制器（resource controllers）：

┌─────────────────────────────────────────────────────────────────┐
│  cpu控制器：                                                     │
│  ├── cpu.max：限制CPU时间上限                                   │
│  ├── cpu.weight：相对权重（类似 CFS 调度）                      │
│  └── 示例：echo "100000 100000" > cpu.max（限制到10%）          │
├─────────────────────────────────────────────────────────────────┤
│  memory控制器：                                                  │
│  ├── memory.max：内存上限（字节）                               │
│  ├── memory.low：内存下限（触发回收）                           │
│  └── 示例：echo "1G" > memory.max                               │
├─────────────────────────────────────────────────────────────────┤
│  io控制器：                                                      │
│  ├── io.max：最大IOPS和带宽                                     │
│  └── 示例：echo "253:0 rbps=max wbps=max" > io.max             │
├─────────────────────────────────────────────────────────────────┤
│  pids控制器：                                                    │
│  ├── pids.max：允许的最大进程数                                 │
│  └── 防止fork炸弹                                              │
└─────────────────────────────────────────────────────────────────┘

实际演示：限制容器内存

$ mkdir -p /sys/fs/cgroup/user/test-container
$ echo "1G" > /sys/fs/cgroup/user/test-container/memory.max
$ echo $$ > /sys/fs/cgroup/user/test-container/cgroup.procs
$ cat /sys/fs/cgroup/user/test-container/memory.max
1073741824  # 1GB = 1024 * 1024 * 1024

现在这个shell进程及其子进程，最多只能使用1GB内存。
```

#### 容器文件系统：overlayfs的奥秘

```
┌─────────────────────────────────────────────────────────────────┐
│              overlayfs：层次化文件系统                           │
└─────────────────────────────────────────────────────────────────┘

Docker镜像为什么能节省空间？
答案：copy-on-write（写时复制）和层次化存储。

┌─────────────────────────────────────────────────────────────────┐
│  overlayfs三层结构：                                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                     merged（联合挂载）                    │   │
│  │              容器看到的最终文件系统                        │   │
│  └────────────────────────────┬────────────────────────────┘   │
│                               │                                  │
│         ┌─────────────────────┼─────────────────────┐          │
│         │                     │                     │          │
│         ▼                     ▼                     ▼          │
│  ┌─────────────┐      ┌─────────────┐       ┌─────────────┐     │
│  │   upperdir  │      │  workdir   │       │   lowerdir │     │
│  │  （容器层）  │      │ （工作目录）│       │  （镜像层）  │     │
│  │ 可写        │      │  临时文件   │       │  只读       │     │
│  │ /var/lib/...│      │            │       │ /var/lib/...│    │
│  └─────────────┘      └─────────────┘       └─────────────┘     │
│                                                                  │
│  读取文件时：                                                    │
│  ├── 先查upperdir（容器层）                                     │
│  ├── 找不到再查lowerdir（镜像层）                               │
│  └── 找到就返回                                                  │
│                                                                  │
│  写入文件时：                                                    │
│  ├── 在upperdir创建新文件                                        │
│  └── 体现为容器的"修改"                                         │
│                                                                  │
│  删除文件时：                                                    │
│  ├── 在upperdir创建whiteout文件（.wh.xxx）                      │
│  └── "遮蔽"lowerdir中的文件                                     │
└─────────────────────────────────────────────────────────────────┘

为什么镜像能节省空间？

┌─────────────────────────────────────────────────────────────────┐
│  场景：1000个容器都基于同一个ubuntu:22.04镜像                    │
│                                                                  │
│  如果每个容器都复制一份：                                        │
│  └── 1000 × ~80MB = 80GB（巨大浪费）                            │
│                                                                  │
│  使用overlayfs：                                                 │
│  ├── 1000个容器的lowerdir都指向同一个镜像层                     │
│  ├── 每个容器只有自己的upperdir（容器层）                       │
│  ├── 如果容器不修改镜像文件，upperdir为空                       │
│  └── 实际磁盘占用：镜像层 + Σ(各容器的修改)                     │
│                                                                  │
│  如果1000个容器都是"干净"的：                                   │
│  └── 80MB + 1000 × ~0 = ~80MB（几乎不占额外空间）               │
│                                                                  │
│  如果每个容器修改了100MB文件：                                   │
│  └── 80MB + 1000 × 100MB = 约100GB                             │
│                                                                  │
│  这就是容器镜像"增量存储"的奥秘！                               │
└─────────────────────────────────────────────────────────────────┘
```

#### runc、containerd、docker：层层封装的关系

```
┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘

理解这个关系，才能理解为什么K8s可以直接使用containerd：

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│                    Docker Engine (dockerd)                       │
│                    ├── Docker CLI (docker)                       │

┌─────────────────────────────────────────────────────────────────┐

│                    ├── 镜像管理                                 │

│                    └── 可以独立管理runC                         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    runc                                          │

└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘

早期K8s（1.24之前）：
├── dockershim：K8s内置的Docker CRI实现
├── Docker Engine比containerd重（包含构建、编排等功能）
└── K8s只需要运行时，不需要构建

K8s 1.24+（现在）：
├── 移除了dockershim
├── 直接使用containerd（通过CRI）
├── 更轻量、更少依赖
└── 为什么？—— Docker build的结果仍然是OCI镜像，containerd能运行

$ ctr --address /run/containerd/containerd.sock \
      images pull docker.io/library/ubuntu:22.04

$ ctr --address /run/containerd/containerd.sock \
      run --rm -t docker.io/library/ubuntu:22.04 test

containerd直接运行Docker镜像，因为两者都遵循OCI标准！
```

### 1.3.1 对称加密 vs 非对称加密

**先理解两种加密的本质区别：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    对称加密（Symmetric Encryption）              │
└─────────────────────────────────────────────────────────────────┘

原理：加密和解密用同一个密钥

┌─────────────────────────────────────────────────────────────────┐
│  Alice                          Bob                              │
│    │                              │                               │
│    │    共享密钥：secret123       │                               │
│    │ ←─────────────────────────→ │                               │
│    │                              │                               │
│    │    加密：明文 + 密钥 → 密文  │                               │
│    │ ──────────────────────────→ │                               │
│    │                              │                               │
│    │    解密：密文 + 密钥 → 明文  │                               │
│    │ ←─────────────────────────── │                               │
│                                                                  │
│  算法：AES、DES、3DES、ChaCha20                                  │
│  特点：                                                          │
│  ├──速度快（比非对称快100-1000倍）                               │
│  ├──密钥短（128位、256位）                                       │
│  └──问题：密钥传输——如何安全传递密钥给对方？                    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                  非对称加密（Asymmetric Encryption）              │
└─────────────────────────────────────────────────────────────────┘

原理：有一对密钥——公钥和私钥

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  数学关系：                                                      │
│  ├── 公钥加密的信息，只有私钥能解密                               │
│  ├── 私钥加密的信息，只有公钥能解密                               │
│  └── 数学上的"陷门函数"——正向容易，逆向困难                     │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    RSA算法原理                           │    │
│  │                                                         │    │
│  │  1. 选择两个大素数 p 和 q                                │    │
│  │  2. 计算 n = p × q                                      │    │
│  │  3. 计算 φ(n) = (p-1) × (q-1)                         │    │
│  │  4. 选择 e 与 φ(n) 互质（通常65537）                    │    │
│  │  5. 计算 d = e⁻¹ mod φ(n)                             │    │
│  │                                                         │    │
│  │  公钥：(n, e)                                           │    │
│  │  私钥：(n, d)                                           │    │
│  │                                                         │    │
│  │  加密：C = Mᵉ mod n                                     │    │
│  │  解密：M = Cᵈ mod n                                     │    │
│  │                                                         │    │
│  │  安全性基于：因式分解大数的困难性                        │    │
│  │  2048位RSA：当前需要数十年才能破解                       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    ECDH算法原理                          │    │
│  │                                                         │    │
│  │  椭圆曲线离散对数问题（更高效）：                        │    │
│  │  Q = d × P                                             │    │
│  │  已知Q和P，求d是困难的                                  │    │
│  │                                                         │    │
│  │  256位ECDH安全性 ≈ 2048位RSA                           │    │
│  │  Google优先使用P-256曲线                                │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3.2 TLS握手：一步步的加密协商过程

**这是理解HTTPS的核心。让我用最详细的方式解释：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    TLS 1.2 完整握手过程                          │
│              （浏览器到GCP负载均衡器为例）                        │
└─────────────────────────────────────────────────────────────────┘

阶段1：TCP连接建立（3次握手）
─────────────────────────────────────────────────────────────────

Client                              Server (GCP LB)
  │                                    │
  │────── SYN ────────────────────────→│  1. 发起连接
  │                                    │
  │←──── SYN-ACK ─────────────────────│  2. 服务器确认
  │                                    │
  │────── ACK ───────────────────────→│  3. 客户端确认
  │                                    │
  │      TCP连接建立完成               │


阶段2：TLS握手（关键！）
────────────────────────────────────────────────────────────────

Step 1: Client Hello（客户端打招呼）

Client ──────────────────────────────────────────────────────────→ Server
      │
      │  发送内容：
      │  ├── TLS版本：TLS 1.2
      │  ├── 支持的加密套件（按优先级排序）：
      │  │   ├── TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
      │  │   ├── TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
      │  │   └── TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
      │  ├── 支持的椭圆曲线：
      │  │   ├── secp256r1 (P-256) ← Google优选
      │  │   ├── secp384r1 (P-384)
      │  │   └── secp521r1 (P-521)
      │  ├── 客户端随机数（Client Random，32字节）
      │  └── Session ID（如果是恢复会话）
      │


Step 2: Server Hello（服务器回应）

Client ←────────────────────────────────────────────────────────── Server
      │
      │  回应内容：
      │  ├── 选择的加密套件：TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
      │  │                          ↑ ECDHE密钥交换
      │  │                          ↑ RSA签名
      │  │                          ↑ AES-128加密
      │  │                          ↑ GCM模式（认证加密）
      │  │                          ↑ SHA-256哈希
      │  ├── 选择的椭圆曲线：secp256r1
      │  ├── 服务器随机数（Server Random，32字节）
      │  └── Session ID


Step 3: Certificate（服务器证书）

Client ←────────────────────────────────────────────────────────── Server
      │
      │  发送内容：X.509证书链
      │
      │  ┌─────────────────────────────────────────────────────┐
      │  │                   证书链结构                         │
      │  │                                                     │
      │  │  Root CA（根证书）                                   │
      │  │  ├── 操作系统/浏览器内置                           │
      │  │  ├── DigiCert Global Root G2                       │
      │  │  └── 有效期：约20年                                 │
      │  │                                                     │
      │  │           ↓                                         │
      │  │                                                     │
      │  │  Intermediate CA（中间证书）                        │
      │  │  ├── GCP证书中间签发机构                            │
      │  │  ├── "GTS CA 1D4" 或 "GTS Root CA"                │
      │  │  └── 有效期：约5年                                  │
      │  │                                                     │
      │  │           ↓                                         │
      │  │                                                     │
      │  │  Server Certificate（服务器证书）                   │
      │  │  ├── CN = *.loadbalancer.google.com                │
      │  │  ├── SAN = 多个域名                                 │
      │  │  ├── 公钥（用于加密）                               │
      │  │  ├── 签名（中间CA用私钥签的）                       │
      │  │  └── 有效期：约3个月 ← 为什么这么短？               │
      │  │                                                     │
      │  └─────────────────────────────────────────────────────┘
      │


Step 4: Server Key Exchange（ECDHE参数）

Client ←────────────────────────────────────────────────────────── Server
      │
      │  发送内容（ECDHE密钥交换所需）：
      │  ├── 椭圆曲线参数（curve secp256r1）
      │  ├── 服务器ECDH公钥（ServerDHParams）
      │  └── 签名：用服务器私钥对Server Random + Client Random + DH参数签名
      │      （证明这些参数确实是服务器发的）


Step 5: Client Key Exchange（客户端密钥交换）

Client ──────────────────────────────────────────────────────────→ Server
      │
      │  发送内容：
      │  └── 客户端ECDH公钥（ClientDHParams）
      │
      │  关键：此时双方都有了对方的公钥


阶段3：密钥派生（关键计算！）
────────────────────────────────────────────────────────────────

双方现在可以独立计算出相同的"主密钥"（Master Secret）：

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  ECDHE密钥交换原理：                                             │
│                                                                  │
│  数学基础：                                                       │
│  ├── 选择椭圆曲线上的一个基点 G                                  │
│  ├── Alice选择私钥 dA，计算公钥 QA = dA × G                     │
│  ├── Bob选择私钥 dB，计算公钥 QB = dB × G                       │
│  ├── Alice计算共享密钥：S = dA × QB = dA × (dB × G)           │
│  ├── Bob计算共享密钥：S = dB × QA = dB × (dA × G)             │
│  └── 两者结果相同！因为椭圆曲线乘法满足交换律                   │
│                                                                  │
│  ══════════════════════════════════════════════════════════════  │
│                                                                  │
│  实际TLS中的计算（双方独立进行）：                               │
│                                                                  │
│  1. 生成Pre-Master Secret（预备主密钥）                         │
│     ┌────────────────────────────────────────────────────────┐   │
│     │  ECDHE: S = dClient × QServer                         │   │
│     │  （客户端用客户端私钥 × 服务器ECDH公钥）               │   │
│     │  或者：S = dServer × QClient                           │   │
│     │  （两者结果相同！）                                     │   │
│     └────────────────────────────────────────────────────────┘   │
│                                                                  │
│  2. 生成Master Secret（主密钥）                                  │
│     ┌────────────────────────────────────────────────────────┐   │
│     │  Master Secret = PRF(                                 │   │
│     │      Pre-Master Secret,                               │   │
│     │      "master secret",                                 │   │
│     │      Client Random + Server Random                    │   │
│     │  )                                                    │   │
│     │                                                        │   │
│     │  PRF = Pseudo-Random Function（伪随机函数）            │   │
│     │  结果：48字节的Master Secret                           │   │
│     └────────────────────────────────────────────────────────┘   │
│                                                                  │
│  3. 派生会话密钥                                                 │
│     ┌────────────────────────────────────────────────────────┐   │
│     │  从Master Secret派生：                                 │   │
│     │                                                        │   │
│     │  ├── client_write_MAC_key[20]    ← HMAC密钥（完整性） │   │
│     │  ├── server_write_MAC_key[20]                         │   │
│     │  ├── client_write_key[16]       ← AES密钥（加密）     │   │
│     │  ├── server_write_key[16]                             │   │
│     │  ├── client_write_IV[16]        ← 初始向量            │   │
│     │  └── server_write_IV[16]                             │   │
│     └────────────────────────────────────────────────────────┘   │
│                                                                  │
│  为什么前向保密（Forward Secrecy）重要？                          │
│  ┌────────────────────────────────────────────────────────┐      │
│  │                                                         │      │
│  │  如果没有ECDHE（只用RSA交换密钥）：                      │      │
│  │  ├── 服务器私钥泄露后，可以解密所有历史会话             │      │
│  │  └── 攻击者可能已经记录了加密流量                       │      │
│  │                                                         │      │
│  │  使用ECDHE后：                                          │      │
│  │  ├── 每次会话的Pre-Master Secret是临时生成的           │      │
│  │  ├── ECDHE私钥是临时生成的（会话结束后丢弃）            │      │
│  │  ├── 服务器长期私钥泄露，无法解密历史会话               │      │
│  │  └── 这就是"前向保密"（Perfect Forward Secrecy）        │      │
│  │                                                         │      │
│  └────────────────────────────────────────────────────────┘      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘


阶段4：加密通信开始
────────────────────────────────────────────────────────────────

Client                              Server
  │                                    │
  │  双方现在都有：                      │
  │  ├── Master Secret                │
  │  ├── client_write_key             │
  │  ├── server_write_key             │
  │  └── MAC keys                     │
  │                                    │
  │  后续所有通信都用这些密钥加密        │
  │  （对称加密，速度快）               │
  │                                    │
  │  完整的加密记录（Record）：         │
  │  ┌────────────────────────────┐    │
  │  │  明文数据                  │    │
  │  │  ↓                        │    │
  │  │  压缩（可选）              │    │
  │  │  ↓                        │    │
  │  │  添加MAC（完整性校验）     │    │
  │  │  ↓                        │    │
  │  │  AES加密                   │    │
  │  │  ↓                        │    │
  │  │  添加TLS头                 │    │
  │  │  ↓                        │    │
  │  │  传输                      │    │
  │  └────────────────────────────┘    │
  │                                    │
```

### 1.3.3 TLS 1.3的改进：为什么更快更安全？

```
TLS 1.2 vs TLS 1.3 握手对比：

┌─────────────────────────────────────────────────────────────────┐
│              TLS 1.2 握手（2-RTT）                              │
│                                                                  │
│  Round Trip 1:                                                  │
│  Client ────── Client Hello ──────────────────────────→ Server  │
│                                                                  │
│               ←──── Server Hello                                 │
│               ←──── Certificate                                 │
│               ←──── Server Key Exchange                          │
│               ←──── Server Hello Done                          │
│                                                                  │
│  Round Trip 2:                                                  │
│  Client ────── Client Key Exchange ──────────────────→ Server  │
│               ───── Change Cipher Spec                          │
│               ───── Finished                                    │
│                                                                  │
│               ←──── Change Cipher Spec                          │
│               ←──── Finished                                    │
│                                                                  │
│  总计：2个RTT + 证书传输                                         │
│  耗时：约100-300ms                                               │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│              TLS 1.3 握手（1-RTT）                               │
│                                                                  │
│  Round Trip 1:                                                  │
│  Client ────── Client Hello ──────────────────────────→ Server  │
│               + supported_cipher_suites                         │
│               + key_share (client DH params)                    │
│                                                                  │
│               ←──── Server Hello                                 │
│               ←──── key_share (server DH params)                │
│               ←──── Certificate                                 │
│               ←──── Certificate Verify                          │
│               ←──── Finished                                    │
│                                                                  │
│               ───── Finished                                    │
│                                                                  │
│  总计：1个RTT                                                    │
│  耗时：约50-100ms（减半！）                                      │
│                                                                  │
│  关键改进：                                                      │
│  ├── 1-RTT握手（0-RTT可选但有重放风险）                         │
│  ├── 移除RSA密钥交换（只有ECDHE）                               │
│  ├── 移除MD5/SHA1等不安全算法                                   │
│  ├── 强制前向保密                                              │
│  └── 简化加密套件（只有5个可用套件）                            │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3.5 常见误区和面试问题

```
┌─────────────────────────────────────────────────────────────────┐
│                    HTTPS常见误区                                  │
└─────────────────────────────────────────────────────────────────┘

❌ 误区1："HTTPS加密后什么都看不到"
   ✓ 事实：只有内容加密，域名、IP、端口可见
   ✓ HTTPS防止的是中间人看到内容，不是隐藏连接存在

❌ 误区2："HTTPS就是SSL"
   ✓ 事实：SSL已废弃（SSL 3.0），TLS是继任者
   ✓ 正确的说法："HTTPS over TLS"

❌ 误区3："HTTPS不影响性能"
   ✓ 事实：握手增加延迟，但现代硬件影响很小
   ✓ TLS 1.3 1-RTT vs 0-RTT可进一步优化
   ✓ HTTP/2多路复用减少连接开销

❌ 误区4："自签名证书也安全"
   ✓ 事实：自签名证书无法证明身份
   ✓ 除非客户端明确信任该证书，否则中间人攻击仍然可能
   ✓ 测试环境可用，生产必须用受信任CA签发的证书

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

面试问题：如果客户端证书过期，HTTPS连接会怎样？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

答案：
1. 客户端证书用于mTLS（双向TLS认证）
2. 服务器验证客户端证书时，如果证书过期：
   - 验证会失败
   - 服务器发送Alert消息
   - 连接终止

场景：API服务要求客户端提供证书
├── 客户端证书过期
├── 服务器验证失败
└── 返回403 Forbidden或400 Bad Request
```

## 1.4 本章小结：理解这些的意义

学完本节，你理解了什么？

```
┌─────────────────────────────────────────────────────────────────┐
│                    从物理到虚拟化的完整链路                       │
└─────────────────────────────────────────────────────────────────┘

        ↓
虚拟化层：
```

### 2.2.2 CIDR深入理解：从"类"到"无类"

**CIDR（Classless Inter-Domain Routing）是你必须掌握的基础。**

```
┌─────────────────────────────────────────────────────────────────┐
│              IP地址"类"的历史                                   │
└─────────────────────────────────────────────────────────────────┘

早期IP分类（已经废弃，但概念仍有用）：

┌─────────────────────────────────────────────────────────────────┐
│  A类：0.0.0.0   - 127.255.255.255   (/8，1677万主机）         │
│  B类：128.0.0.0 - 191.255.255.255   (/16，6.5万主机）         │
│  C类：192.0.0.0 - 223.255.255.255   (/24，254主机）           │
│  D类：224.0.0.0 - 239.255.255.255   （多播，无主机位）        │
│  E类：240.0.0.0 - 255.255.255.255   （保留，实验用）           │
└─────────────────────────────────────────────────────────────────┘

问题：每个B类网络6.5万主机，对于大多数组织来说浪费严重

解决：CIDR（1993年）

┌─────────────────────────────────────────────────────────────────┐
│              CIDR表示法：网络前缀/前缀长度                        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  10.0.0.0/8  是什么意思？                                        │
│                                                                  │
│  /8表示：前8位是网络标识，后24位是主机标识                       │
│                                                                  │
│  二进制视角：                                                    │
│  00001010.00000000.00000000.00000000                            │
│  │──────││───────────────│                                     │
│  网络ID    主机ID（全部0）                                        │
│                                                                  │
│  可用主机数：2^24 - 2 = 16,777,214                              │
│  （减去网络地址和广播地址）                                       │
│                                                                  │
│  10.0.0.0/8 的含义：                                            │
│  ├── 网络地址：10.0.0.0                                         │
│  ├── 广播地址：10.255.255.255                                  │
│  ├── 第一个可用IP：10.0.0.1                                    │
│  └── 最后一个可用IP：10.255.255.254                            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

常见CIDR换算（必须熟练）：

┌─────────────────────────────────────────────────────────────────┐
│  /24 = 256个地址（254可用）    相当于一个C类                     │
│  /16 = 65,536个地址            相当于一个B类                      │
│  /8  = 16,777,216个地址       相当于一个A类                      │
│                                                                  │
│  /28 = 16个地址（14可用）     适合小型子网                       │
│  /29 = 8个地址（6可用）       适合点对点链接                     │
│  /30 = 4个地址（2可用）       适合路由器之间直连                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2.3 GCP VPC IP地址规划最佳实践

```
┌─────────────────────────────────────────────────────────────────┐
│              GCP VPC IP地址规划策略                              │
└─────────────────────────────────────────────────────────────────┘

推荐VPC IP空间划分（RFC 1918私有地址）：

┌─────────────────────────────────────────────────────────────────┐
│  RFC 1918私有地址范围：                                         │
│  ├── 10.0.0.0/8    （1677万地址）← 推荐                        │
│  ├── 172.16.0.0/12 （104万地址）                               │
│  └── 192.168.0.0/16 （6.5万地址）                             │
└─────────────────────────────────────────────────────────────────┘

大型组织GCP IP规划示例：

VPC：10.0.0.0/16 （约6.5万地址）

├── 子网：prod-web    10.0.1.0/24  → 254台Web服务器
├── 子网：prod-app    10.0.2.0/24  → 254台应用服务器
├── 子网：prod-db     10.0.3.0/24  → 254台数据库服务器
├── 子网：prod-cache  10.0.4.0/24  → 254台缓存服务器

为什么/16而不是/8？
├── /16足够大（6.5万地址）
├── /8太大，可能与其他系统冲突（很多公司用10.0.0.0/8）
└── /16在GCP中足够大多数用例
```

**理解这些特殊地址在排查问题时很有用：**

```
┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘

│  └── 防火墙规则中的"所有地址"                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 2.3 路由表：数据包的导航系统

### 2.3.1 路由的本质

**路由表就是数据包的GPS导航。让我用类比解释：**

```
┌─────────────────────────────────────────────────────────────────┐
│              路由表：数据包的GPS导航                             │
└─────────────────────────────────────────────────────────────────┘

现实场景：你在北京，要去上海开会

你的"路由表"：
┌─────────────────────────────────────────────────────────────────┐
│  目的地              │  下一跳          │  说明                   │
├─────────────────────────────────────────────────────────────────┤
│  上海（本地）        │  直接到达        │  你在上海本地           │
│  上海市内           │  出租车          │  目的在上海市内        │
│  北京以外           │  机场/火车站     │  需要离开北京           │
│  其他国家           │  国际机场        │  需要出国               │
└─────────────────────────────────────────────────────────────────┘

网络中的路由表完全一样：

┌─────────────────────────────────────────────────────────────────┐
│  目的地（Destination）│  下一跳（Next Hop）  │  说明              │
├─────────────────────────────────────────────────────────────────┤
│  10.0.1.0/24         │  Local            │  本地子网，直接交付  │
│  10.0.2.0/24         │  10.0.0.1        │  通过内部路由        │
│  0.0.0.0/0           │  203.0.113.1    │  默认路由（外网）    │
│  172.16.0.0/12       │  10.0.0.254      │  对等连接VPC        │
└─────────────────────────────────────────────────────────────────┘

数据包如何匹配路由？

┌─────────────────────────────────────────────────────────────────┐
│  路由匹配规则：最长前缀匹配（Longest Prefix Match）               │
│                                                                  │
│  数据包目的IP：10.0.2.100                                        │
│                                                                  │
│  路由表：                                                        │
│  ├── 0.0.0.0/0         → 默认路由，匹配0位                     │
│  ├── 10.0.0.0/16       → 匹配16位                              │
│  └── 10.0.2.0/24        → 匹配24位 ✓（最长匹配）                │
│                                                                  │
│  结果：数据包发送到10.0.2.0/24对应的下一跳                        │
│                                                                  │
│  为什么这样设计？                                                │
│  ├── 更具体的路由优先                                           │
│  ├── 可以有精细的流量控制                                       │
│  └── 默认路由作为兜底                                           │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3.2 GCP VPC路由的工作原理

```
│              GCP路由优先级（数值越小优先级越高）                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  优先级对比（数值越小越优先）：                                   │
│                                                                  │
│  1. 实例级别路由（NIC路由）  → 优先级最高（100）                 │
│     - VM有多个网卡时，每个网卡可以有不同路由                     │
│                                                                  │
│  2. 子网路由                → 100                               │
│     - 指向同一VPC内其他子网的流量                               │
│                                                                  │
│  3. VPC对等连接路由         → 200                               │
│     - 通过VPC Peering连接的VPC                                   │
│                                                                  │
│  4. 负载均衡器路由          → 300                               │
│     - GCP内部负载均衡器                            │
│                                                                  │
│  5. VPN路由                 → 400-500                          │
│     - Cloud VPN隧道                                 │
│                                                                  │
│  6. 默认路由（0.0.0.0/0）   → 500-900（可配置）                │
│     - 访问互联网的流量                              │
│                                                                  │
│  7. 互联网网关              → 800（最低优先级）                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

实际例子：多层路由选择

┌─────────────────────────────────────────────────────────────────┐
│  场景：VM在子网10.0.1.0/24，访问10.0.2.100                      │
│                                                                  │
│  数据包：目的IP = 10.0.2.100                                     │
│                                                                  │
│  路由表按优先级检查：                                            │
│                                                                  │
│  Step 1: 检查所有路由的掩码长度                                  │
│                                                                  │
│  可用路由：                                                      │
│  ├── 10.0.0.0/8     → Local                                   │
│  ├── 10.0.1.0/24    → Local（VM所在子网）                     │
│  ├── 10.0.2.0/24    → <某个实例> ← 最长匹配！                  │
│  └── 0.0.0.0/0      → 默认网关                                │
│                                                                  │
│  Step 2: 选择最长匹配                                            │
│  ├── 0.0.0.0/0      匹配0位                                   │
│  ├── 10.0.0.0/8     匹配8位                                   │
│  ├── 10.0.1.0/24    匹配24位                                  │
│  └── 10.0.2.0/24    匹配24位 ← 两者都匹配24位，优先级相同？   │
│                                                                  │
│  Step 3: 如果有两个相同长度的路由（罕见情况）                   │
│  └── GCP会负载均衡到两个下一跳                                  │
│                                                                  │
│  结论：数据包发送到10.0.2.100的下一跳                            │
└─────────────────────────────────────────────────────────────────┘
```

## 2.4 防火墙：网络安全的核心

### 2.4.1 防火墙的本质：包过滤

**防火墙不是魔法，它只是一个根据规则过滤数据包的"检查站"。**

```
┌─────────────────────────────────────────────────────────────────┐
│              防火墙工作原理：包过滤（Packet Filtering）           │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│                    数据包                                        │
│                      │                                           │
│                      ▼                                           │
│            ┌─────────────────┐                                   │
│            │   防火墙规则     │                                   │
│            │                 │                                   │
│            │ 检查：           │                                   │
│            │ - 源IP           │                                   │
│            │ - 目的IP         │                                   │
│            │ - 协议           │                                   │
│            │ - 端口           │                                   │
│            │ - 动作（ALLOW/   │                                   │
│            │   DENY）         │                                   │
│            └────────┬────────┘                                   │
│                     │                                            │
│           ┌────────┴────────┐                                   │
│           ▼                 ▼                                   │
│      ALLOW               DENY                                   │
│    放行数据包           丢弃数据包                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

防火墙检查的五元组：

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  源IP（Source IP）      │  192.168.1.100                       │
│  目的IP（Dest IP）      │  10.0.2.50                           │
│  协议（Protocol）       │  TCP                                  │
│  源端口（Source Port）  │  54321（随机高端口）                 │

│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  五元组组合决定一个"连接"                                 │    │
│  │  (192.168.1.100, 54321, 10.0.2.50, 443, TCP)            │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  注意：防火墙通常不检查源端口，只检查目的端口                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4.2 GCP防火墙规则详解

**GCP防火墙规则比传统防火墙强大得多：**

```
┌─────────────────────────────────────────────────────────────────┐
│              GCP防火墙规则结构                                   │
└─────────────────────────────────────────────────────────────────┘

GCP防火墙规则组成：

┌─────────────────────────────────────────────────────────────────┐

┌─────────────────────────────────────────────────────────────────┐
│  1. 方向（Direction）                                           │
│                                                                  │
│  INGRESS（入站）：控制进入VPC的流量                             │
│  ├── 数据包从外部进入VM                                         │
│  └── 常见场景：允许用户访问Web服务                              │
│                                                                  │
│  EGRESS（出站）：控制离开VPC的流量                              │
│  ├── 数据包从VM发送到外部                                       │

┌─────────────────────────────────────────────────────────────────┐
│  2. 来源/目标（Source / Destination）                           │
│                                                                  │
│  入站规则（INGRESS）可以指定：                                   │
│  ├── sourceTags：源VM的标签                                    │
│  ├── sourceServiceAccounts：源服务账号                          │
│  ├── sourceRanges：IP范围（如0.0.0.0/0表示所有）               │
│  └── sametadata：服务账号元数据                                │
│                                                                  │
│  出站规则（EGRESS）可以指定：                                   │
│  ├── destTags：目标VM的标签                                     │
│  ├── destServiceAccounts：目标服务账号                          │

│  └── destinationags：目标协议（如https）                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  3. 协议和端口（Protocol and Port）                             │
│                                                                  │
│  可以指定：                                                      │
│  ├── tcp:22（SSH）                                             │
│  ├── tcp:80,443（HTTP/HTTPS）                                  │
│  ├── tcp:0-65535（所有TCP）                                    │
│  ├── udp:53（DNS）                                             │
│  ├── icmp（ping）                                              │
│  └── all（所有协议）                                            │
│                                                                  │
│  快捷写法：                                                      │
│  ├── tcp:22,80,443,3000-4000                                  │
│  └── 可以混合指定多个端口和范围                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4.3 防火墙优先级：数字越小越重要

```
┌─────────────────────────────────────────────────────────────────┐
│              GCP防火墙优先级系统（0-65535）                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  优先级对比（数字越小，优先级越高）：                             │
│                                                                  │
│  0-99     → 最高优先级（系统保留）                               │
│  100-499  → 高优先级（用户定义）                                 │
│  500-999  → 中优先级                                            │
│  1000+    → 低优先级                                            │
│  65535    → 最低优先级（默认deny）                              │
└─────────────────────────────────────────────────────────────────┘

隐式规则：

┌─────────────────────────────────────────────────────────────────┐
│  入站（INGRESS）：                                              │
│  ├── 隐式DENY：没有任何规则匹配的入站流量被拒绝                  │
│  └── 原因：安全默认，拒绝所有不明流量                           │
│                                                                  │
│  出站（EGRESS）：                                               │
│  ├── 隐式ALLOW：没有任何规则匹配的出站流量被允许                │
│  └── 原因：方便出站通信，但如果需要严格控制应该创建出站DENY规则 │
└─────────────────────────────────────────────────────────────────┘

常见场景：优先级设计

┌─────────────────────────────────────────────────────────────────┐
│  场景：允许办公网络SSH，拒绝其他所有SSH                          │
│                                                                  │
│  规则1：allow-ssh-from-office                                   │
│  ├── 优先级：100（高）                                          │
│  ├── 方向：INGRESS                                             │
│  ├── 源IP：203.0.113.0/24（办公网络）                          │
│  ├── 协议端口：tcp:22                                          │
│  └── 动作：ALLOW                                                │
│                                                                  │
│  规则2：deny-ssh-all（显式拒绝）                               │
│  ├── 优先级：65534                                             │
│  ├── 方向：INGRESS                                             │
│  ├── 源IP：0.0.0.0/0                                          │
│  ├── 协议端口：tcp:22                                          │
│  └── 动作：DENY                                                │
│                                                                  │
│  结果：办公网络可以SSH，其他IP无法SSH                           │
│  为什么不只创建allow规则？                                       │
│  └── 显式拒绝更清晰，也方便日志追踪                            │
└─────────────────────────────────────────────────────────────────┘
```

## 2.5 Cloud NAT：让私有VM上网的原理

### 2.5.1 为什么私有IP的VM需要NAT？

```
┌─────────────────────────────────────────────────────────────────┐
│              私有IP的限制                                        │
└─────────────────────────────────────────────────────────────────┘

私有IP地址（RFC 1918）不能在互联网上路由：

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  VM（私有IP 10.0.1.100） ──────→ 互联网                         │
│       │                                                          │
│       └── 发送数据包：源IP=10.0.1.100                          │
│                                                                  │
│  互联网服务器收到后，不知道往哪回复：                            │
│  ├── 10.0.1.100不是公网可路由的地址                             │
│  └── 路由器会丢弃这个数据包                                     │
│                                                                  │
│  解决方案：NAT（网络地址转换）                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.5.4 NAT vs 公网IP：成本和安全比较

```
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  成本对比：                                                      │
│                                                                  │
│  公网IP（静态）：$0.008/小时 = $5.76/月                         │
│  Cloud NAT：$0.044/GB处理的数据                                │
│                                                                  │
│  场景：100台VM，每台每月用10GB流量                              │
│                                                                  │
│  方案A：每台都有公网IP                                          │
│  └── 成本：100 × $5.76 = $576/月                               │
│                                                                  │
│  方案B：私有IP + Cloud NAT                                      │
│  └── 成本：100 × 10GB × $0.044 = $44/月                        │
│                                                                  │
│  节省：$532/月（92%！）                                         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  安全对比：                                                      │
│                                                                  │
│  公网IP的问题：                                                  │
│  ├── 直接暴露在互联网上                                        │
│  ├── 需要配置防火墙规则防止未授权访问                           │
│  ├── 可能被扫描和攻击                                          │
│  └── 需要额外的DDoS防护                                         │
│                                                                  │
│  NAT的优势：                                                     │
│  ├── VM不直接暴露                                              │
│  ├── 只有出站连接可以建立                                      │
│  ├── 外部无法主动连接到私有VM                                   │
│  └── 减少攻击面                                               │
│                                                                  │
│  但NAT不是万能的：                                              │
│  ├── 无法接受入站连接（除非通过端口转发）                      │
│  ├── 某些协议可能不兼容NAT（需要NAT穿透）                      │
│  └── 游戏服务器等需要公网IP的场景不适用                       │
└─────────────────────────────────────────────────────────────────┘
```

## 3.1 为什么需要IAM？——从"所有人都能操作"到"最小权限"

### 3.1.1 不用IAM的风险

```
│                                                                  │
│  IAM解决方案：                                                   │
│  ├── 每个用户有自己的账号                                       │

└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐

│                                                                  │
│  IAM解决方案：                                                   │
│  ├── 开发者只有VM操作权限                                       │

│  └── 最小权限原则                                               │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐

│                                                                  │
│  服务账号密钥JSON文件：                                          │
│  ├── 被提交到GitHub（常见错误）                                 │

│                                                                  │

│  ├── 使用Workload Identity（不暴露密钥）                        │
```

## 3.2 IAM核心概念：Who-Can-Do-What-OnWhich

### 3.2.1 三要素详解

**每个IAM权限问题都可以分解为这三个要素：**

```
┌─────────────────────────────────────────────────────────────────┐
│              IAM三要素                                           │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   WHO（谁）        │  CAN DO WHAT（能做什么）│  ON WHICH RESOURCE（在什么资源上）│
│        │          │            │            │           │              │
│        ▼          │            ▼            │           ▼              │
│   Principal       │         Role           │        Resource          │
│   （身份）        │       （权限集合）      │       （资源）            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  WHO（身份类型）：                                               │
│                                                                  │
│  1. Google Account                                              │
│     user:alice@example.com                                      │
│     真实用户，人类操作                                          │
│                                                                  │
│  2. Service Account                                             │
│     service-account:vm-writer@myproject.iam.gserviceaccount.com │
│     机器账号，程序操作                                          │
│                                                                  │
│  3. Google Group                                                │
│     group:developers@example.com                                │
│     用户集合，方便批量管理                                       │
│                                                                  │
│  4. Cloud Identity Domain                                       │
│     通过企业IdP管理                                              │
│                                                                  │
│  5. All authenticated users                                    │
│     任何有效Google账号                                           │
│                                                                  │
│  6. All users                                                   │
│     任何人（包括匿名）                                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2.2 Role vs Permission：最容易混淆的概念

```
┌─────────────────────────────────────────────────────────────────┐
│              Role和Permission的关系                             │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Permission（权限）：最小单元                                     │
│  ├── 格式：service.resource.verb                                │
│  │   例如：compute.instances.create                             │
│  │   └── compute服务 + instances资源 + create动作              │
│  ├── 描述"能做什么"                                            │
│  └── 不能单独授予，必须打包成Role                               │
│                                                                  │
│  Role（角色）：权限集合                                          │
│  ├── 一个Role包含多个Permission                                 │
│  ├── 三种类型：Primitive、Predefined、Custom                   │
│  └── 授予Principal时实际上授予的是Role                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │   Role = { Permission1, Permission2, Permission3, ... }  │   │
│  │                                                         │   │
│  │   例子：roles/compute.instanceAdmin                     │   │
│  │   ├── compute.instances.get                            │   │
│  │   ├── compute.instances.list                           │   │
│  │   ├── compute.instances.create                         │   │
│  │   ├── compute.instances.start                          │   │
│  │   ├── compute.instances.stop                          │   │
│  │   └── ... (约20个相关权限)                              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2.3 三种Role类型对比

```
┌─────────────────────────────────────────────────────────────────┐
│              IAM Role类型详解                                    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  1. Primitive Roles（基本角色）- 历史遗留，不推荐                 │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  roles/owner                                            │   │
│  │  ├── 所有权限                                            │   │
│  │  ├── 包括删除项目和billing                               │   │
│  │  └── 不应该分配给任何人                                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  roles/editor                                           │   │
│  │  ├── 修改所有资源                                        │   │
│  │  ├── 不能管理IAM和billing                               │   │
│  │  └── 生产环境禁用！                                     │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  roles/viewer                                           │   │
│  │  ├── 只读所有资源                                        │   │
│  │  └── 可用于审计账号                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  问题：权限太大，不够精细                                        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  2. Predefined Roles（预定义角色）- 推荐                        │
│                                                                  │
│  GCP为每种服务预定义的细粒度角色：                                │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  roles/compute.instanceAdmin                           │   │
│  │  ├── 管理VM实例                                        │   │
│  │  ├── 创建/删除/启动/停止                               │   │
│  │  ├── 不能管理网络和防火墙                              │   │
│  │  └── 不能删除持久磁盘（保护数据）                      │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  roles/compute.storageAdmin                           │   │
│  │  ├── 管理磁盘和快照                                    │   │
│  │  └── 不能删除持久磁盘（需要单独权限）                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  roles/compute.networkAdmin                           │   │
│  │  ├── 管理网络资源                                      │   │
│  │  ├── 不能修改防火墙（需要compute.securityAdmin）        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  优点：细粒度，专职专用                                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  3. Custom Roles（自定义角色）- 按需创建                        │
│                                                                  │
│  当预定义角色权限不够或太多时：                                   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  场景：需要只读VM状态，但不能看日志                      │   │
│  │                                                         │   │
│  │  预定义角色问题：                                       │   │
│  │  ├── roles/compute.viewer → 权限太多（所有只读）       │   │
│  │  └── 组合roles不够灵活                                  │   │
│  │                                                         │   │
│  │  解决方案：创建Custom Role                              │   │
│  │  ├── compute.instances.get                            │   │
│  │  ├── compute.instances.list                           │   │
│  │  ├── compute.instances.getSerialPortOutput            │   │
│  │  └── compute.instances.status                         │   │
│  │                                                         │   │
│  │  不包含：                                               │   │
│  │  ├── logging.views.list（排除日志读取）                │   │
│  │  └── monitoring.timeSeries.list（排除指标读取）        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  限制：不能使用在组织级别（只能用在该项目）                       │
└─────────────────────────────────────────────────────────────────┘
```

## 3.3 IAM Policy结构与评估逻辑

### 3.3.1 IAM Policy的JSON结构

**理解Policy结构是理解权限评估的基础：**

```
┌─────────────────────────────────────────────────────────────────┐
│              IAM Policy JSON结构                                │
└─────────────────────────────────────────────────────────────────┘

{
  "bindings": [
    {
      "role": "roles/compute.instanceAdmin",
      "members": [
        "user:alice@example.com",
        "serviceAccount:vm-writer@myproject.iam.gserviceaccount.com",
        "group:developers@example.com"
      ]
    },
    {
      "role": "roles/storage.objectViewer",
      "members": [
        "user:bob@example.com"
      ]
    }
  ],
  "etag": "BwUjM2hMX9Y=",
  "version": 1
}

┌─────────────────────────────────────────────────────────────────┐
│  字段解释：                                                      │
│                                                                  │
│  bindings：权限绑定数组                                          │
│  ├── role：Role ID                                              │
│  └── members：拥有这个Role的身份列表                            │
│                                                                  │
│  members格式：                                                   │
│  ├── user:<email>           Google账号                         │
│  ├── serviceAccount:<email> 服务账号                           │
│  ├── group:<email>          Google群组                         │
│  ├── domain:<domain>       整个域                             │
│  └── allUsers / allAuthenticatedUsers                         │
│                                                                  │
│  etag：版本控制，用于并发更新冲突检测                            │
│  version：Policy版本（0, 1, 3）                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3.2 权限评估逻辑：DENY优先

**这是IAM最核心也最容易出错的部分：**

```
┌─────────────────────────────────────────────────────────────────┐
│              IAM权限评估流程                                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│                    访问请求到达                                  │
│                        │                                         │
│                        ▼                                         │
│                  ┌───────────┐                                   │
│                  │ 检查DENY？ │ ← 第一优先级                      │
│                  └─────┬─────┘                                   │
│                        │                                          │
│            ┌───────────┴───────────┐                            │
│            ▼                       ▼                            │
│       有DENY ──────────────────→ 结束：DENY（拒绝）              │
│            │                                                      │
│            否                                                     │
│            │                                                      │
│            ▼                                                      │
│     ┌───────────┐                                                │
│     │ 检查ALLOW？│                                                │
│     └─────┬─────┘                                                │
│           │                                                      │
│           ▼                                                      │
│    ┌────────────┐    ┌────────────┐                              │
│    │ 有ALLOW？  │───→│ 结束：DENY │（隐式拒绝）                  │
│    └─────┬──────┘    └────────────┘                              │
│          │                                                       │
│          是                                                      │
│          │                                                       │
│          ▼                                                       │
│    结束：ALLOW（允许）                                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

关键点1：DENY永远优先

┌─────────────────────────────────────────────────────────────────┐
│  场景：用户alice同时拥有以下权限：                               │
│                                                                  │
│  ├── roles/compute.editor（允许创建VM）                        │
│  └── 项目级别DENY：禁示create instances                        │
│                                                                  │
│  结论：DENY优先，alice无法创建VM                                │
└─────────────────────────────────────────────────────────────────┘

关键点2：没有ALLOW就是DENY（隐式拒绝）

┌─────────────────────────────────────────────────────────────────┐
│  场景：用户bob没有任何明确授予的权限：                           │
│                                                                  │
│  结果：bob无法执行任何GCP操作（隐式DENY）                        │
│                                                                  │
│  默认安全：GCP的默认行为是拒绝                                   │
└─────────────────────────────────────────────────────────────────┘

关键点3：评估范围是项目级别

┌─────────────────────────────────────────────────────────────────┐
│  场景：在项目A有权限，在项目B没有权限                            │
│                                                                  │
│  项目A：alice有roles/compute.admin                              │
│  项目B：alice没有任何角色                                       │
│                                                                  │
│  结果：alice可以在项目A管理VM，但在项目B什么都不能做             │
│                                                                  │
│  项目级别的权限隔离是GCP的基本原则                               │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3.3 多绑定叠加规则

```
┌─────────────────────────────────────────────────────────────────┐
│              多Role叠加规则                                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  场景：alice拥有多个Role                                         │
│                                                                  │
│  Role 1: roles/compute.instanceAdmin                             │
│  Role 2: roles/storage.objectViewer                             │

┌─────────────────────────────────────────────────────────────────┐
│  场景：多个条件绑定同一Principal                                  │
│                                                                  │
│  Binding 1:                                                     │
│  ├── role: roles/compute.viewer                                 │
│  └── member: user:alice@example.com                             │
│                                                                  │
│  Binding 2:                                                      │
│  ├── role: roles/compute.instanceAdmin                          │
│  └── member: user:alice@example.com                             │
│                                                                  │
│  结果：alice拥有两个Role的并集                                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  场景：不同成员绑定同一Role                                       │
│                                                                  │
│  Binding:                                                        │
│  ├── role: roles/compute.admin                                  │
│  └── members:                                                    │
│      ├── user:alice@example.com                                 │
│      ├── user:bob@example.com                                   │
│      └── group:team@example.com                                 │
│                                                                  │
│  结果：alice、bob、team组内所有人都拥有compute.admin              │
└─────────────────────────────────────────────────────────────────┘
```

## 3.4 服务账号：机器身份的原理

### 3.4.1 服务账号 vs 用户账号

```
┌─────────────────────────────────────────────────────────────────┐
│              服务账号的本质                                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  用户账号（User Account）：                                       │
│  ├── 代表真实的人                                               │
│  ├── 需要密码登录                                               │
│  ├── 用于人类操作GCP                                           │
│  └── 适合：管理员、开发人员、运维                               │
│                                                                  │
│  服务账号（Service Account）：                                    │
│  ├── 代表运行的程序/机器                                        │
│  ├── 不需要密码，使用密钥或Workload Identity认证                  │
│  ├── 用于程序操作GCP                                           │
│  └── 适合：VM、Cloud Functions、CI/CD pipeline                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  为什么服务账号是安全的？                                        │
│                                                                  │
│  1. 最小权限：程序只需要需要的权限                              │
│  2. 可追溯：操作记录的是服务账号，不是用户                      │
│  3. 可管理：可以随时禁用或删除服务账号                          │
│  4. 无密码：避免密码泄露风险                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4.2 服务账号认证方式演进

```
┌─────────────────────────────────────────────────────────────────┐
│              服务账号认证方式对比                                │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  方式1：服务账号密钥（JSON）- 不推荐                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  工作原理：                                             │   │
│  │  ├── 创建服务账号时生成RSA密钥对                       │   │
│  │  ├── 公钥存储在Google，私钥下载为JSON                   │   │
│  │  ├── 程序使用私钥生成JWT token                         │   │
│  │  └── Google验证JWT获取访问令牌                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  风险：                                                          │
│  ├── 私钥文件泄露 = 永久权限丢失                                │
│  ├── 密钥可能上传到GitHub                                      │
│  ├── 无法限制使用IP范围                                        │
│  └── Google会在90天后提醒轮换                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  方式2：附加到VM的服务账号 - 推荐（GCE场景）                     │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  工作原理：                                             │   │
│  │  ├── 创建VM时指定Service Account                       │   │
│  │  ├── GCP自动为VM提供credentials                       │   │
│  │  ├── VM内部的程序无需额外配置                          │   │
│  │  ├── GCP自动轮换访问令牌                               │   │
│  │  └── 实际上是Metadata Server代理的                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  优势：                                                          │
│  ├── 无需管理密钥文件                                          │
│  ├── GCP自动管理令牌轮换                                       │
│  ├── 可以限制OAuth Scope                                       │
│  └── VM启动时自动获取凭证                                      │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  方式3：Workload Identity（K8s/GKE场景）- 最佳                   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  工作原理：K8s ServiceAccount ↔ GCP ServiceAccount映射   │   │
│  │                                                         │   │
│  │                                                         │   │
│  │  Pod想访问GCP资源：                                      │   │
│  │  1. Pod使用K8s ServiceAccount token                     │   │
│  │  2. Kubernetes向GCP STS服务请求                         │   │
│  │  3. GCP返回GCP服务账号的访问令牌                        │   │
│  │  4. Pod使用GCP令牌访问GCP资源                           │   │
│  │                                                         │   │
│  │  关键：Pod永远拿不到GCP私钥                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  优势：                                                          │
│  ├── 密钥不进入集群环境                                         │
│  ├── 权限可以在GCP层面精细控制                                  │
│  ├── 可以审计谁在什么时候访问了什么                             │
│  └── 支持跨项目访问                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4.3 Workload Identity详解（面试常问）

```
┌─────────────────────────────────────────────────────────────────┐
│              Workload Identity工作流程                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                      GCP IAM                              │   │
│  │                                                         │   │
│  │    K8s ServiceAccount    ↔    GCP ServiceAccount          │   │
│  │    (sa-name)                 (gcp-sa@project.iam.gservice│   │
│  │                                 account.com)             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│                              │ 绑定：允许K8s SA扮演GCP SA        │
│                              ▼                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    GKE集群                               │   │
│  │                                                         │   │
│  │    ┌─────────────────────────────────────────────┐     │   │
│  │    │                  Pod                        │     │   │
│  │    │                                            │     │   │
│  │    │  1. Pod使用K8s SA token                     │     │   │
│  │    │  2. 请求GCP STS (Security Token Service)    │     │   │
│  │    │  3. 获取GCP access token                    │     │   │
│  │    │  4. 使用token访问GCP API                    │     │   │
│  │    │                                            │     │   │
│  │    └─────────────────────────────────────────────┘     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

配置步骤：

1. 创建GCP Service Account
   gcloud iam service-accounts create gcp-sa \
       --project=my-project

2. 授予GCP SA需要的权限
   gcloud projects add-iam-policy-binding my-project \
       --member="serviceAccount:gcp-sa@my-project.iam.gserviceaccount.com" \
       --role="roles/bigquery.dataViewer"

3. 建立K8s SA和GCP SA的映射
   gcloud iam service-accounts add-iam-policy-binding \
       gcp-sa@my-project.iam.gserviceaccount.com \
       --role="roles/iam.workloadIdentityUser" \
       --member="serviceAccount:my-project.svc.id.goog[default/my-app-sa]"

4. 在K8s中标注SA
   kubectl annotate serviceaccount my-app-sa \
       iam.gke.io/gcp-service-account=gcp-sa@my-project.iam.gserviceaccount.com
```

### 3.5.2 资源层级权限继承

```
┌─────────────────────────────────────────────────────────────────┐
│              GCP资源层级与权限继承                               │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Organization（组织）←─── 在这层授予的权限会向下传递            │
│      │                                                            │
│      ├── Folder（文件夹）←── 部门A的权限不传给部门B              │

│      │           │                                               │

│      │                                                            │

│            │                                                      │
│            └── Project B                                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

权限继承规则：
├── Organization → Folder → Project：权限向下传递
├── Folder之间：权限不共享
├── Project之间：权限不共享
└── 资源级别：只继承来自Project的权限

示例：Project级别的Role授予

│  在Project A授予alice roles/compute.admin：                      │
│                                                                  │
│  结果：alice可以管理Project A的所有计算资源                      │
│  但alice不能访问Project B的任何资源                             │
│                                                                  │
│  示例：Folder级别的Role授予                                       │
│                                                                  │
│  在Engineering Folder授予bob roles/viewer：                      │
│                                                                  │
│  结果：bob可以查看Folder内所有Project的资源                      │
│  包括Project A、Project B、Project C                             │
└─────────────────────────────────────────────────────────────────┘
```

```
# VPC网络 + 子网 + Firewall Rules + NAT Gateway
# 文件：vpc.tf

# ─────────────────────────────────────────────────────────────────
# VPC网络
# ─────────────────────────────────────────────────────────────────
resource "google_compute_network" "main" {
  name                    = "main-vpc"
  auto_create_subnetworks = false  # 手动创建子网
  description             = "主VPC网络"
}

# ─────────────────────────────────────────────────────────────────
# 子网配置
# ─────────────────────────────────────────────────────────────────
resource "google_compute_subnetwork" "subnets" {
  for_each = {
    prod-web    = "10.0.1.0/24"
    prod-app    = "10.0.2.0/24"
    prod-db     = "10.0.3.0/24"
    prod-cache  = "10.0.4.0/24"
    dev-web     = "10.0.10.0/24"
    dev-app     = "10.0.11.0/24"
    mgmt        = "10.0.100.0/24"
  }

  name          = each.key
  network       = google_compute_network.main.id
  ip_cidr_range = each.value
  region        = "us-central1"

  private_ip_google_access = true  # 允许私有IP访问Google API

  log_config {
    aggregation_interval = "INTERVAL_10_MIN"
    flow_sampling        = 0.5
    metadata             = "INCLUDE_ALL_METADATA"
  }
}

# ─────────────────────────────────────────────────────────────────
# Firewall Rules
# ─────────────────────────────────────────────────────────────────

# 允许健康检查入站
resource "google_compute_firewall" "allow_health_checks" {
  name    = "allow-health-checks"
  network = google_compute_network.main.name

  source_ranges = ["130.211.0.0/22", "35.191.0.0/16"]
  target_tags   = ["http-server", "https-server"]

  allow {
    protocol = "tcp"
    ports    = ["80", "443"]
  }
}

# 允许内部网络通信
resource "google_compute_firewall" "allow_internal" {
  name    = "allow-internal"
  network = google_compute_network.main.name

  source_tags = ["allow-internal"]

  allow {
    protocol = "tcp"
    ports    = ["0-65535"]
  }
  allow {
    protocol = "udp"
    ports    = ["0-65535"]
  }
  allow {
    protocol = "icmp"
  }
}

# 允许SSH（仅从管理子网）
resource "google_compute_firewall" "allow_ssh_from_mgmt" {
  name    = "allow-ssh-from-mgmt"
  network = google_compute_network.main.name

  source_ranges = ["10.0.100.0/24"]
  target_tags   = ["ssh-enabled"]

  allow {
    protocol = "tcp"
    ports    = ["22"]
  }
}

# 显式拒绝所有SSH（低优先级）
resource "google_compute_firewall" "deny_ssh_all" {
  name    = "deny-ssh-all"
  network = google_compute_network.main.name

  priority = 65534  # 低优先级

  source_ranges = ["0.0.0.0/0"]

  deny {
    protocol = "tcp"
    ports    = ["22"]
  }
}

# ─────────────────────────────────────────────────────────────────
# Cloud NAT
# ─────────────────────────────────────────────────────────────────
resource "google_compute_router" "nat_router" {
  name    = "nat-router"
  network = google_compute_network.main.id
  region  = "us-central1"
}

resource "google_compute_address" "nat_ip" {
  name         = "nat-ip"
  region       = "us-central1"
  address_type = "EXTERNAL"
  network_tier = "PREMIUM"
}

resource "google_compute_router_nat" "main" {
  name                               = "main-nat"
  router                             = google_compute_router.nat_router.name
  region                             = "us-central1"
  nat_ip_allocate_option             = "MANUAL_ONLY"
  nat_ips                            = [google_compute_address.nat_ip.self_link]
  source_subnetwork_ip_ranges_to_nat = "ALL_SUBNETWORKS"

  log_config {
    filter = "ERRORS_ONLY"
  }
}
```

```
设计决策背后的原理：

1. 为什么用Global LB而不是Regional LB？
   ├── 全球用户访问同一个IP，自动路由到最近区域
   ├── 提高用户体验（降低延迟）
   └── 故障时自动切换到其他区域

2. 为什么Web层不用Spot VM？
   ├── Web层直接面对用户，需要保证可用性
   ├── Spot可能在任何时候被收回
   └── 用户请求中断会影响体验

3. 为什么App层可以用Spot？
   ├── App层有多个实例，单个故障不影响整体
   ├── Stateless应用，可以快速重新启动
   ├── 成本敏感，可以节省60%费用
   └── 配合自动扩缩容，可用性与成本平衡

4. 为什么用Regional Internal LB？
   ├── 区域内流量不收费（节省成本）
   ├── 比Global Internal LB便宜
   └── 区域内延迟更低

5. 为什么Database用Regional？
   ├── 数据库需要低延迟
   ├── 跨区域复制有延迟和成本
   └── 写操作必须在主节点进行
```

### B.2 微服务架构设计：GKE vs Cloud Run

**什么时候该用GKE？什么时候该用Cloud Run？**

```
┌─────────────────────────────────────────────────────────────────┐
│              GKE vs Cloud Run 决策树                            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  开始                                                             │
│    │                                                             │
│    ▼                                                             │
│  是否需要长期运行的服务？                                        │
│    │                                                             │
│    ├── 否 ─────────────────→ Cloud Run（适合事件驱动）           │
│    │                                                             │
│    └── 是 → 需要K8s高级特性？                                   │
│              ├── 否 → Cloud Run（简化运维）                    │
│              │                                                    │
│              └── 是 → 需要哪些特性？                            │
│                      ├── HPA（水平自动扩缩）→ 两者都支持        │
│                      ├── VPA（垂直扩缩）→ GKE                  │
│                      ├── DaemonSet → GKE                        │
│                      ├── StatefulSet → GKE                     │
│                      ├── Custom Metrics → GKE                   │
│                      └── Ingress → Cloud Run                    │
│                              │                                   │
│                              ▼                                   │
│                          GKE（完整K8s能力）                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  成本对比：                                                      │
│                                                                  │
│  Cloud Run：                                                     │
│  ├── 按请求计费（$0.40/million请求）                            │
│  ├── 闲置实例不收费（冷启动）                                   │
│  └── 适合间歇性工作负载                                        │
│                                                                  │
│  GKE：                                                          │
│  ├── 按节点计费（E2实例约$0.05/vCPU-小时）                     │
│  ├── 无论是否处理请求，节点都要收费                             │
│  └── 适合持续高负载、需要K8s特性的场景                          │
│                                                                  │
│  成本分界点估算：                                               │
│  假设单个Pod配置：2 vCPU, 1GB内存                              │
│                                                                  │
│  Cloud Run成本：                                                │
│  - 处理100万请求，假设每次请求100ms                             │
│  - 100万 × 0.1秒 = 100,000 CPU-秒                             │
│  - ≈ 28 CPU-小时 × $0.40/千CPU-小时 ≈ $0.01                   │
│                                                                  │
│  GKE成本：                                                      │
│  - 2 vCPU × 24小时 × 30天 = 1,440 vCPU-小时                   │
│  - 1,440 × $0.05 = $72/月                                      │
│                                                                  │
│  结论：                                                          │
│  - 如果请求量小（< 1万/天）→ Cloud Run更便宜                   │
│  - 如果请求持续不断 → GKE更便宜                                 │
└─────────────────────────────────────────────────────────────────┘
```

### B.4 灾难恢复架构设计

**如何设计一个能容忍区域级故障的架构？**

```
┌─────────────────────────────────────────────────────────────────┐
│              多区域灾难恢复架构                                  │
└─────────────────────────────────────────────────────────────────┘

核心设计原则：

1. 数据复制
   ├── 同步复制：适合低RPO（0）
   │   └── Cloud SQL同步复制到备用区域
   └── 异步复制：适合可接受数据丢失
       └── BigQuery快照复制、GCS双区域

2. 流量切换
   ├── DNS failover：Cloud DNS with health check
   ├── 预设备用区域IP
   └── 自动化脚本或Terraform切换

3. 容量规划
   ├── 备用区域预留最小容量
   ├── 故障时快速扩容
   └── 平时可以使用Spot降低空闲成本

┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│                    Primary Region (us-central1)                 │
│                         │                                        │
│                         ▼                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │   │
│  │  │  MIG(3)     │  │  MIG(3)     │  │  MIG(3)     │      │   │
│  │  │  ZONE A     │  │  ZONE B     │  │  ZONE C     │      │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘      │   │
│  │                                                         │   │
│  │  Cloud SQL (Regional) ←── Sync Replication ──→           │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         │                                        │
│                         │ Async Replication                     │
│                         ▼                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Backup Region (us-east1)                                │   │
│  │                                                         │   │
│  │  ┌─────────────┐  ┌─────────────┐                       │   │
│  │  │  MIG(min 1) │  │  MIG(min 1) │                       │   │
│  │  │  Zone A     │  │  Zone B     │                       │   │
│  │  └─────────────┘  └─────────────┘                       │   │
│  │       │                                    │                │   │
│  │       │  Cold standby: scale to 0         │                │   │
│  │       │  when primary healthy             │                │   │
│  │       ▼                                    ▼                │   │
│  │  ┌─────────────┐  ┌─────────────┐                       │   │
│  │  │ Cloud SQL    │  │ GCS Bucket  │                       │   │
│  │  │ (Standby)   │  │ (Multi-region)│                     │   │
│  │  └─────────────┘  └─────────────┘                       │   │
│  │                                                         │   │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘

RPO/RTO目标与实现：

┌─────────────────────────────────────────────────────────────────┐
│  RPO (Recovery Point Objective) - 能容忍丢失多少数据？           │
│                                                                  │
│  RPO = 0（零数据丢失）：                                        │
│  ├── Cloud SQL with synchronous replication                    │
│  ├── Zonal PD with replication                                 │
│  └── 成本：高                                                  │
│                                                                  │
│  RPO = 1小时：                                                  │
│  ├── BigQuery定时快照到备用区域                                 │
│  ├── GCS Object Lifecycle到双区域                              │
│  └── 成本：中                                                   │
│                                                                  │
│  RPO = 24小时：                                                 │
│  ├── 每日备份到备用区域                                         │
│  └── 成本：低                                                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  RTO (Recovery Time Objective) - 故障后多久恢复？               │
│                                                                  │
│  RTO = 分钟级：                                                  │
│  ├── Multi-region MIG with pre-warmed instances               │
│  ├── Cloud DNS自动failover                                     │
│  └── 需要完整的自动化切换脚本                                   │
│                                                                  │
│  RTO = 小时级：                                                 │
│  ├── Regional服务降级到备用区域                                 │
│  ├── 手动或半自动切换                                          │
│  └── 成本：中等                                                 │
│                                                                  │
│  RTO = 天级：                                                   │
│  ├── 从备份恢复                                                │
│  └── 成本：低                                                   │
└─────────────────────────────────────────────────────────────────┘
```
