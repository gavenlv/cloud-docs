# 云架构设计

## 本章概述

架构设计是云技术专家的核心能力。本章将学习高可用架构、灾难恢复、弹性伸缩等高级架构设计。

## 学习目标

- 掌握高可用架构设计原则
- 理解灾难恢复策略
- 学会弹性伸缩设计
- 掌握无服务器架构
- 理解事件驱动架构
- 学会数据湖架构设计

### 1.2 可用性计算

```
可用性等级

99%      (两个9)    年停机时间: 3.65天
99.9%    (三个9)    年停机时间: 8.77小时
99.99%   (四个9)    年停机时间: 52.6分钟
99.999%  (五个9)    年停机时间: 5.26分钟
99.9999% (六个9)    年停机时间: 31.5秒

可用性计算公式：
可用性 = MTBF / (MTBF + MTTR)

MTBF: 平均故障间隔时间
MTTR: 平均修复时间
```

## 2. 灾难恢复策略

### 2.1 DR策略类型

```
灾难恢复策略对比

┌──────────────┬─────────────┬─────────────┬───────────────────────────┐
│    策略      │   RTO       │    RPO      │         说明              │
├──────────────┼─────────────┼─────────────┼───────────────────────────┤
│ Backup&Restore│   小时级    │   天级      │ 成本最低，恢复最慢        │
├──────────────┼─────────────┼─────────────┼───────────────────────────┤
│ Pilot Light  │   分钟级    │   分钟级    │ 核心服务预热，快速启动    │
├──────────────┼─────────────┼─────────────┼───────────────────────────┤
│ Warm Standby │   分钟级    │   秒级      │ 缩减版备用环境持续运行    │
├──────────────┼─────────────┼─────────────┼───────────────────────────┤
│ Multi-Site   │   秒级      │   秒级      │ 全量多活，成本最高        │
└──────────────┴─────────────┴─────────────┴───────────────────────────┘

RTO (Recovery Time Objective): 恢复时间目标
RPO (Recovery Point Objective): 恢复点目标

  - type: Resource

      name: cpu

        type: Utilization

### 5.2 数据湖架构深度原理

**数据湖是怎么实现海量数据存储和处理的？**

```
┌─────────────────────────────────────────────────────────────────┐
│              数据湖核心机制解析                                     │
└─────────────────────────────────────────────────────────────────┘

数据分层架构：

┌─────────────────────────────────────────────────────────────────┐
│  数据湖分层设计：                                                 │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Raw Zone (Bronze Layer)                               │   │
│  │  ├── 原始数据，未经处理                                  │   │
│  │  ├── 保持数据完整性                                      │   │
│  │  ├── 按摄入时间分区                                      │   │
│  │  └── 示例：s3://lake/raw/2024/03/19/                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Clean Zone (Silver Layer)                             │   │
│  │  ├── 清洗后的数据                                        │   │
│  │  ├── 标准化格式（Parquet/ORC）                          │   │
│  │  ├── 去重、过滤、转换                                    │   │
│  │  └── 示例：s3://lake/clean/users/                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Curated Zone (Gold Layer)                              │   │
│  │  ├── 业务就绪的数据                                      │   │
│  │  ├── 聚合、汇总、计算                                    │   │
│  │  ├── 优化查询性能                                        │   │
│  │  └── 示例：s3://lake/curated/daily_sales/              │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

列式存储原理：

┌─────────────────────────────────────────────────────────────────┐
│  Parquet列式存储：                                               │
│                                                                  │
│  行式存储（传统数据库）：                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Row 1: [id:1, name:Alice, age:25, city:NYC]            │   │
│  │  Row 2: [id:2, name:Bob, age:30, city:LA]               │   │
│  │  Row 3: [id:3, name:Carol, age:28, city:NYC]            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  列式存储（Parquet）：                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Column id:    [1, 2, 3]                               │   │
│  │  Column name:  [Alice, Bob, Carol]                     │   │
│  │  Column age:   [25, 30, 28]                            │   │
│  │  Column city:  [NYC, LA, NYC]                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  优势：                                                          │
│  ├── 只读取需要的列（减少I/O）                                   │
│  ├── 同一列数据类型相同，压缩率高                                │
│  ├── 适合分析查询（OLAP）                                        │
│  └── 支持谓词下推（过滤在读取前执行）                            │   │
└─────────────────────────────────────────────────────────────────┘

分区与分桶：

┌─────────────────────────────────────────────────────────────────┐
│  数据组织策略：                                                   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 分区（Partitioning）                                  │   │
│  │  ├── 按日期分区：year=2024/month=03/day=19              │   │
│  │  ├── 按地区分区：country=us/state=ca                    │   │
│  │  ├── 优点：查询时只扫描相关分区                           │   │
│  │  └── 缺点：小文件问题                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  2. 分桶（Bucketing）                                    │   │
│  │  ├── 按哈希值分桶：hash(id) % 10                         │   │
│  │  ├── 固定桶数：10个桶                                    │   │
│  │  ├── 优点：避免数据倾斜                                  │   │
│  │  └── 优化JOIN操作                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  示例目录结构：                                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  s3://lake/clean/orders/                              │   │
│  │  ├── year=2024/                                       │   │
│  │  │   ├── month=03/                                     │   │

## 6. 实操项目

### 项目：设计高可用电商架构

**架构需求**：
- 可用性目标：99.99%
- RTO：< 15分钟
- RPO：< 5分钟
- 支持自动伸缩
- 多区域灾备

**架构设计**：

```yaml
architecture:
  dns:
    service: Route53
    config:
      health-checks: enabled
      failover: automatic
      
  cdn:
    service: CloudFront
    config:
      origins:
        - alb-primary
        - s3-static-assets
      caching:
        ttl: 86400
        
  load-balancer:
    service: ALB
    config:
      scheme: internet-facing
      availability-zones: 3
      health-check:
        path: /health
        interval: 10s
        
  compute:
    service: EKS
    config:
      node-groups:
        - name: web
          min: 2
          max: 10
          instance-type: m6i.large
        - name: worker
          min: 1
          max: 5
          instance-type: c6i.large
      autoscaling:
        enabled: true
        
  database:
    primary:
      service: Aurora MySQL
      config:
        instance-class: db.r6g.large
        replicas: 2
        multi-az: true
        backup-retention: 7
    cache:
      service: ElastiCache Redis
      config:
        node-type: cache.r6g.large
        replicas: 2
        cluster-mode: enabled
        
  storage:
    service: S3
    config:
      versioning: enabled
      replication: cross-region
      lifecycle:
        - transition-to-ia: 90d
        - transition-to-glacier: 180d
        
  disaster-recovery:
    strategy: warm-standby
    secondary-region: us-west-2
    database-replication: global
    dns-failover: enabled
```
