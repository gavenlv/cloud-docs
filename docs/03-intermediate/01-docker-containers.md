# Docker容器技术

## 本章概述

容器技术是云原生的核心基础。Docker作为最流行的容器化工具，彻底改变了软件的开发、部署和运维方式。本章将深入学习Docker，通过详尽的原理分析和命令解析，帮助你不仅知道"怎么做"，更理解"为什么这样做"。

## 学习目标

- 理解Docker架构原理和底层技术
- 掌握Dockerfile每条指令的含义和最佳实践
- 深入理解镜像优化原理
- 全面掌握Docker网络机制
- 深入理解Docker存储原理
- 熟练使用Docker Compose

## 1. Docker架构原理

### 1.1 Docker是什么？为什么需要它？

```
传统部署方式 vs Docker容器部署

│                                                                         │

│  ├── 依赖冲突: 不同项目需要不同版本的Python/Node等                       │
│  ├── 环境差异: 开发机Windows，生产机Linux                               │
│  ├── 配置文件: 路径、密码等在不同环境不同                               │
│  ├── 资源争用: 多个服务在同一服务器上的冲突                             │

│                                                                         │

│                                                                         │
│  优势：                                                                │
│  ├── 一致性: "一次构建，到处运行"                                       │

│  ├── 轻量: 共享操作系统内核，启动快，资源占用少                        │
│  ├── 可移植: 任何支持Docker的机器都能运行                               │

│                                                                         │
```

### 1.4 底层技术 - Namespaces和Cgroups

```
Docker底层技术 - 容器隔离的原理

为什么要了解底层技术？
├── 排查问题: 当容器出现问题时，需要理解根本原因
├── 优化性能: 知道资源是如何被管理的
└── 安全加固: 理解隔离机制才能更好地保护容器

=======================================================================
Namespaces (命名空间) - 隔离系统资源
=======================================================================

PID Namespace (进程隔离)
┌─────────────────────────────────────────────────────────────────┐
│  宿主机进程列表:                                                  │
│  ├── PID 1: systemd                                             │
│  ├── PID 2: kthreadd                                            │
│  ├── ...                                                        │
│  ├── PID 100: dockerd                                           │
│  └── PID 101: containerd                                        │
│                                                                   │
│  容器内进程列表 (独立PID空间):                                     │
│  ├── PID 1: /bin/bash (容器内的第一个进程)                       │
│  ├── PID 2: nginx                                               │
│  ├── PID 3: python app.py                                        │
│  └── ...                                                         │
│                                                                   │
│  原理: 容器内的进程看不到宿主机和其他容器的进程                      │
│  为什么需要: 进程隔离让容器感觉自己是唯一的进程                      │
└─────────────────────────────────────────────────────────────────┘

NET Namespace (网络隔离)
┌─────────────────────────────────────────────────────────────────┐
│  容器拥有独立的:                                                   │
│  ├── 网络接口 (eth0, lo等)                                       │
│  ├── IP地址                                                       │
│  ├── 路由表                                                       │
│  └── 端口空间                                                     │
│                                                                   │
│  原理: 每个容器有独立的网络栈                                       │
│  为什么需要:                                                     │
│  ├── 不同容器可以监听相同端口而不冲突                             │
│  └── 容器间的网络通信需要明确配置                                 │
└─────────────────────────────────────────────────────────────────┘

IPC Namespace (进程间通信隔离)
┌─────────────────────────────────────────────────────────────────┐
│  隔离:                                                            │
│  ├── 信号量                                                       │
│  ├── 共享内存                                                     │
│  └── 消息队列                                                    │
│                                                                   │
│  原理: 容器内的进程通信不影响其他容器                               │
└─────────────────────────────────────────────────────────────────┘

MNT Namespace (挂载点隔离)
┌─────────────────────────────────────────────────────────────────┐
│  隔离:                                                            │
│  ├── 文件系统挂载点                                               │
│  └── 根目录 (/)                                                  │
│                                                                   │
│  原理: 容器内的文件系统是独立的                                   │
│  为什么需要:                                                     │
│  ├── 容器内的文件系统操作不影响宿主机和其他容器                    │
│  └── 可以挂载不同的卷到容器的相同路径                             │
└─────────────────────────────────────────────────────────────────┘

UTS Namespace (主机名隔离)
┌─────────────────────────────────────────────────────────────────┐
│  隔离:                                                            │
│  ├── 主机名 (hostname)                                           │
│  └── 域名 (domainname)                                           │
│                                                                   │
│  原理: 容器可以有自己的主机名                                      │
└─────────────────────────────────────────────────────────────────┘

USER Namespace (用户隔离)
┌─────────────────────────────────────────────────────────────────┐
│  隔离:                                                            │
│  └── 用户ID和组ID                                                 │
│                                                                   │
│  原理: 容器内的root (UID 0)可能映射到宿主机上的非root用户          │
│  为什么需要: 增加安全性，防止容器内root获得宿主机root权限           │
└─────────────────────────────────────────────────────────────────┘

=======================================================================
Cgroups (控制组) - 资源限制
=======================================================================

为什么需要资源限制？
├── 防止失控: 防止单个容器耗尽系统资源
├── 公平调度: 确保资源在多个容器间合理分配
└── 隔离保障: 一个容器的问题不影响其他容器

CPU限制
┌─────────────────────────────────────────────────────────────────┐
│  语法: --cpus="1.5" 或 --cpuset-cpus="0,1"                      │
│                                                                   │
│  场景:                                                            │
│  ├── --cpus="0.5": 最多使用0.5个CPU核心                          │
│  ├── --cpus="1.5": 最多使用1.5个CPU核心                          │
│  └── --cpuset-cpus="0,3": 只使用第0和第3个CPU核心                 │
│                                                                   │
│  实际效果: 容器内的进程不能超过指定的CPU使用                        │
└─────────────────────────────────────────────────────────────────┘

内存限制
┌─────────────────────────────────────────────────────────────────┐
│  语法: -m=512m 或 --memory=1g                                     │
│                                                                   │
│  场景:                                                            │
│  ├── -m=256m: 最多使用256MB内存                                  │
│  ├── --memory-reservation=128m: 软限制128MB                     │
│  └── --memory-swap: 允许使用多少Swap                             │
│                                                                   │
│  OOM Killer: 当内存超限时，Linux会杀掉容器进程                    │
└─────────────────────────────────────────────────────────────────┘

IO限制
┌─────────────────────────────────────────────────────────────────┐
│  语法: --device-read-bps/--device-write-bps                      │
│                                                                   │
│  场景:                                                            │
│  ├── --device-read-bps=/dev/sda:1mb 限制读带宽                   │
│  └── --device-write-iops=/dev/sda:10 限制写IOPS                  │
└─────────────────────────────────────────────────────────────────┘

网络限制
┌─────────────────────────────────────────────────────────────────┐
│  语法: --limit-rate                                              │
│                                                                   │
│  场景:                                                            │
│  └── 限制容器网络带宽                                            │
└─────────────────────────────────────────────────────────────────┘

=======================================================================
UnionFS (联合文件系统) - 分层存储
=======================================================================

为什么需要分层？
├── 空间节省: 相同层可以共享
├── 缓存复用: 构建时可以使用之前缓存的层
└── 快速部署: 只需下载新的层

原理图解：
┌─────────────────────────────────────────────────────────────────┐
│                                                                   │
│   Layer 4: Container Layer (读写层)                              │
│   ├── 容器运行时的修改                                            │
│   └── 新建/修改/删除文件                                         │
│   ════════════════════════════════════════════════════════════   │
│   Layer 3: CMD/ENTRYPOINT                                        │
│   ════════════════════════════════════════════════════════════   │
│   Layer 2: COPY src                                              │
│   ════════════════════════════════════════════════════════════   │
│   Layer 1: RUN pip install                                       │
│   ════════════════════════════════════════════════════════════   │
│   Layer 0: FROM python:3.11                                      │
│                                                                   │
│   写时复制 (Copy-on-Write):                                      │
│   当需要修改底层文件时，会先复制到上层再修改                        │
└─────────────────────────────────────────────────────────────────┘
```

## 2. Dockerfile最佳实践

### 2.1 Dockerfile指令详解 - 每条指令的原理

```dockerfile
# Dockerfile 每条指令详解
# 为什么要这样写？每个参数是什么意思？

# ============================================================
# FROM - 基础镜像选择
# ============================================================
# 作用: 指定构建的基础镜像
# 为什么需要: 容器需要操作系统和运行时环境

# 不推荐: 使用 latest 标签
FROM node:latest  # ❌ 每次构建可能不同，导致不可预测

# 推荐: 使用具体版本
FROM node:18-alpine  # ✅ 每次构建结果一致

# 推荐: 使用官方镜像
FROM python:3.11-slim  # ✅ 官方维护，安全更新
FROM nginx:alpine  # ✅ 轻量级，适合生产

# 各种基础镜像大小对比
# alpine (~5MB) < debian (~25MB) < ubuntu (~77MB) < centos (~200MB+)

# 最佳实践
# ├── 优先使用官方镜像
# ├── 总是指定版本标签
# ├── 生产环境使用 alpine 或 slim 变体
# └── 考虑使用 distroless (仅包含应用)


# ============================================================
# ARG - 构建参数
# ============================================================
# 作用: 定义构建时变量，仅在构建过程中可用
# 为什么需要: 让构建过程可配置

ARG APP_VERSION=1.0.0
ARG BUILD_DATE
# 
# 使用场景: 
# ├── 注入版本号
# ├── 注入Git提交Hash
# └── 注入构建日期
#
# 使用方式: 在后续指令中用 ${ARG_NAME} 引用


# ============================================================
# ENV - 环境变量
# ============================================================
# 作用: 设置容器运行时的环境变量
# 为什么需要: 配置应用的运行环境

ENV NODE_ENV=production \
    PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1
#
# 为什么要设置这些:
# ├── NODE_ENV=production: 告诉框架使用生产模式
# ├── PYTHONUNBUFFERED=1: 日志实时输出，不缓冲
# └── PYTHONDONTWRITEBYTECODE=1: 不生成.pyc文件


# ============================================================
# WORKDIR - 工作目录
# ============================================================
# 作用: 设置后续指令的工作目录
# 为什么需要: 
# ├── 避免使用绝对路径
# └── 确保文件操作在正确目录

WORKDIR /app
WORKDIR /app/src  # 相对路径基于上一个WORKDIR
#
# 最佳实践:
# ├── 使用绝对路径
# ├── 避免频繁切换
# └── 创建不存在的目录


# ============================================================
# COPY vs ADD - 文件复制
# ============================================================
# 作用: 将文件复制到镜像中

# 推荐: 使用 COPY
COPY package*.json /app/  # 复制文件
COPY src/ /app/src/  # 复制目录

# ADD 的特殊功能 (但推荐用 COPY)
ADD http://example.com/file.tar.gz /tmp/  # 可以从URL下载
ADD --chown=user:group file /app/  # 可以指定所有者

# 为什么推荐用 COPY:
# ├── 语义更明确
# ├── 不支持URL下载（安全）
# └── 不支持自动解压


# ============================================================
# RUN - 执行命令
# ============================================================
# 作用: 在构建时执行命令
# 为什么需要: 安装依赖、运行构建脚本等

# 单行 RUN
RUN apt-get update && apt-get install -y curl

# 多行 RUN (推荐)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        nginx \
    && rm -rf /var/lib/apt/lists/*
#
# 为什么这样做:
# ├── --no-install-recommends: 不安装推荐包，减小体积
# ├── && 连接命令: 减少层数
# └── rm -rf: 清理缓存，减少体积


# ============================================================
# EXPOSE - 声明端口
# ============================================================
# 作用: 声明容器监听的端口
# 注意: 这只是文档，实际端口映射由运行时决定

EXPOSE 8080
EXPOSE 80 443
#
# 为什么要声明:
# ├── 文档作用: 告诉使用者需要映射哪些端口
# └── 工具提示: Docker相关工具会显示端口信息
#
# 实际映射: docker run -p 8080:8080


# ============================================================
# USER - 运行用户
# ============================================================
# 作用: 设置后续命令的执行用户
# 为什么需要: 安全考虑，避免以root运行

RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser
#
# 为什么需要非root用户:
# ├── 权限最小化原则
# ├── 防止容器内漏洞影响宿主机
# └── 符合容器安全最佳实践


# ============================================================
# HEALTHCHECK - 健康检查
# ============================================================
# 作用: 告诉Docker如何检查容器健康状态
# 为什么需要: 负载均衡器需要知道哪些容器是健康的

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1
#
# 参数详解:
# ├── --interval=30s: 检查间隔 (默认30s)
# ├── --timeout=3s: 超时时间 (默认30s)
# ├── --start-period=5s: 启动等待时间 (默认0s)
# ├── --retries=3: 连续失败次数 (默认3次)
# └── CMD: 检查命令
#
# 退出码:
# ├── 0: 健康
# └── 1: 不健康


# ============================================================
# CMD vs ENTRYPOINT - 启动命令
# ============================================================
# 作用: 容器启动时执行的命令

# CMD - 默认命令，可被覆盖
CMD ["node", "app.js"]  # exec格式
CMD node app.js  # shell格式

# ENTRYPOINT - 固定入口，不可覆盖
ENTRYPOINT ["node", "app.js"]

# 组合使用 (常见模式)
ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["postgres"]

# 区别:
# ├── CMD: 可以被 docker run 的参数覆盖
# └── ENTRYPOINT: docker run 的参数会作为参数追加
```

### 2.2 多阶段构建详解

```dockerfile
# 多阶段构建 - 原理和优势

# ============================================================
# 第一阶段: 构建环境
# ============================================================
FROM golang:1.21 AS builder

# 在构建阶段安装所有编译依赖
WORKDIR /app

COPY go.mod go.sum ./
RUN go mod download

COPY . .

# 编译二进制文件
# CGO_ENABLED=0: 静态编译，不需要C库
# GOOS=linux: 目标操作系统
# -installsuffix cgo: 排除CGO相关包
RUN CGO_ENABLED=0 GOOS=linux go build \
    -a -installsuffix cgo \
    -o main .

# ============================================================
# 第二阶段: 运行阶段
# ============================================================
# 使用轻量级基础镜像
FROM alpine:3.18

# 只复制必要的运行时依赖
RUN apk --no-cache add ca-certificates tzdata

WORKDIR /root/

# 从构建阶段复制编译好的二进制
COPY --from=builder /app/main .
COPY --from=builder /app/config ./config

# 设置时区
ENV TZ=Asia/Shanghai

EXPOSE 8080

CMD ["./main"]

# 为什么需要多阶段构建？
# 
# ┌─────────────────────────────────────────────────────────────────┐
# │                        优势对比                                  │
# ├─────────────────────────────────────────────────────────────────┤
# │  单阶段构建:                                                     │
# │  ├── 最终镜像包含: 编译器、构建工具、源码、构建缓存              │
# │  ├── 镜像大小: 可能超过1GB                                      │
# │  └── 安全风险: 包含很多不必要的攻击面                          │
# ├─────────────────────────────────────────────────────────────────┤
# │  多阶段构建:                                                     │
# │  ├── 最终镜像只包含: 编译好的二进制和运行时                      │
# │  ├── 镜像大小: 可能只有10-50MB                                  │
# │  └── 安全提升: 最小化攻击面                                     │
# └─────────────────────────────────────────────────────────────────┘
```

### 3.1 镜像分层原理详解

```
镜像分层 - 为什么要这样设计？

┌─────────────────────────────────────────────────────────────────────────┐
│                    镜像分层架构                                        │
│                                                                         │
│  为什么需要分层？                                                       │
│  ├── 共享复用: 相同层的镜像可以共享存储                                 │
│  ├── 构建缓存: 只需要重新构建变化的层                                   │
│  └── 增量传输: 只需要传输变化的层                                       │
│                                                                         │
│  ─────────────────────────────────────────────────────────────────    │
│                                                                         │
│  层结构:                                                               │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │ Layer N: 最后一条指令生成的层                                   │ │
│  │     (CMD/ENTRYPOINT的结果)                                     │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Layer N-1: COPY 生成                                           │ │
│  │     (复制源代码)                                                │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Layer N-2: RUN 生成                                            │ │
│  │     (安装依赖 npm install)                                      │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Layer N-3: COPY 生成                                           │ │
│  │     (复制package.json)                                         │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ ...                                                            │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Layer 1: FROM 生成                                             │ │
│  │     (基础镜像 python:3.11)                                     │ │
│  └─────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  写时复制 (Copy-on-Write) 原理:                                        │
│                                                                         │
│  1. 初始状态:                                                         │
│  ┌──────────────┐    ┌──────────────┐                                │
│  │  镜像层      │    │  容器层      │                                │
│  │  (只读)     │    │  (空)       │                                │
│  └──────────────┘    └──────────────┘                                │
│                                                                         │
│  2. 容器写入文件:                                                     │
│  ┌──────────────┐    ┌──────────────┐                                │
│  │  镜像层      │    │  容器层      │                                │
│  │  (只读)     │───>│  file (副本) │  复制到容器层                 │
│  └──────────────┘    └──────────────┘                                │
│                                                                         │
│  3. 修改后:                                                           │
│  ┌──────────────┐    ┌──────────────┐                                │
│  │  镜像层      │    │  容器层      │                                │
│  │  (不变)     │    │  file (修改) │  只修改容器层                 │
│  └──────────────┘    └──────────────┘                                │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 优化技巧详解

```dockerfile
# ============================================================
# 技巧1: 选择合适的基础镜像
# ============================================================

# 问题: 为什么要选择特定的基础镜像？
# 
# 镜像大小对比:
# ├── scratch: 0KB (空镜像，需要自己添加所有内容)
# ├── alpine: ~5MB (最小化Linux发行版)
# ├── debian:slim: ~25MB
# ├── ubuntu: ~77MB
# ├── centos: ~200MB
# └── python:3.11: ~1GB (包含完整Python环境)

# 生产环境推荐:
FROM python:3.11-slim  # ~140MB，安装依赖后约200-300MB
FROM node:18-alpine    # ~15MB，安装依赖后约50-100MB

# 极度轻量场景:
FROM gcr.io/distroless/python3-debian11  # 只有Python，无shell


# ============================================================
# 技巧2: 减少层数
# ============================================================

# 问题: 为什么要减少层数？
# 
# 层数影响:
# ├── 更多层 = 更大的镜像 (每层都有元数据)
# ├── 更多层 = 更长的构建时间
# └── 更多层 = 更复杂的缓存管理

# 不好: 多个独立的RUN
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y nginx
RUN rm -rf /var/lib/apt/lists/*

# 好: 合并成一个RUN
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        nginx \
    && rm -rf /var/lib/apt/lists/*


# ============================================================
# 技巧3: 利用构建缓存
# ============================================================

# 问题: 为什么要按特定顺序复制文件？
# 
# Docker缓存机制:
# ├── 从上到下执行每条指令
# ├── 如果指令和之前完全相同，使用缓存
# ├── 如果某层变化，后续所有层都需要重建
# └── 变化频繁的放后面

# 不好: 先复制全部代码
COPY . /app
RUN npm install  # 每次代码变化都需要重新安装依赖

# 好: 先复制依赖文件
COPY package*.json /app/
RUN npm install  # 只在依赖变化时重新安装
COPY . /app     # 代码变化不影响依赖安装


# ============================================================
# 技巧4: 清理缓存和临时文件
# ============================================================

# 每个包管理器的清理方式:

# APT (Debian/Ubuntu)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        package1 \
        package2 \
    && rm -rf /var/lib/apt/lists/*

# pip (Python)
RUN pip install --no-cache-dir -r requirements.txt

# npm (Node.js)
RUN npm ci --only=production

# Maven (Java)
RUN mvn package -DskipTests && \
    rm -rf ~/.m2/repository

# yum/dnf (RHEL/CentOS)
RUN dnf install -y package && \
    dnf clean all


# ============================================================
# 技巧5: 使用.dockerignore
# ============================================================

# 问题: 为什么要使用.dockerignore？
# 
# 原因:
# ├── 减小构建上下文
# ├── 加快构建速度
# └── 避免意外复制敏感文件

# .dockerignore 文件示例:
.git              # 版本控制
.gitignore        # 忽略文件
*.md              # 文档文件
node_modules/     # 本地依赖（应该用容器内安装）
npm-debug.log     # npm日志
.env.local        # 本地环境变量
dist/             # 构建输出（应该用容器内构建）
coverage/         # 测试覆盖率
*.log             # 日志文件
.DS_Store         # macOS系统文件
Thumbs.db         # Windows系统文件
.vscode/          # 编辑器配置
.idea/            # IDE配置
```

```dockerfile
# 镜像优化实战 - 一个Go应用的大小对比

# ============================================================
# 方案1: 直接使用 golang 镜像
# ============================================================
FROM golang:1.21
```

### 4.2 网络命令详解

```bash
# ============================================================
# 创建自定义网络
# ============================================================

# 为什么要创建自定义网络？
# ├── 隔离: 不同项目/环境使用不同网络
# ├── DNS: 自动服务发现（容器名=域名）
# └── 网络控制: 可以配置子网、网关等

# 基本创建
docker network create my-network

# 完整参数说明:
docker network create \
    --driver bridge \              # 网络驱动类型 (bridge/overlay/host/none)
    --subnet 172.20.0.0/16 \    # 子网 CIDR 格式
    --gateway 172.20.0.1 \       # 网关 IP 地址
    --ip-range 172.20.5.0/24 \ # 分配给容器的IP范围
    --label "environment=prod" \ # 标签，用于管理
    my-network

# 参数详解:
# ├── --driver: 
# │   ├── bridge: 单主机网络（默认）
# │   ├── overlay: 跨主机网络（Swarm模式）
# │   ├── host: 共享主机网络
# │   └── none: 无网络
# ├── --subnet: 
# │   └── 网络号/前缀长度
# ├── --gateway: 
# │   └── 网络的网关IP
# └── --ip-range: 
#     └── DHCP分配范围


# ============================================================
# 查看网络
# ============================================================

# 列出所有网络
docker network ls

# 查看网络详情 (包含连接的容器)
docker network inspect my-network

# 查看特定容器的网络
docker inspect container_name --format='{{json .NetworkSettings.Networks}}'


# ============================================================
# 连接/断开容器
# ============================================================

# 将容器连接到网络
docker network connect my-network container_name

# 断开容器与网络的连接
docker network disconnect my-network container_name


# ============================================================
# 容器网络配置
# ============================================================

# 指定IP地址启动容器
docker run -d --name web \
    --network my-network \
    --ip 172.20.0.10 \
    nginx

# 参数说明:
# ├── --network: 指定加入的网络
# ├── --ip: 指定固定IP地址（需要自定义网络）
# ├── --dns: 自定义DNS服务器
# ├── --hostname: 容器内部主机名
# └── --add-host: 添加 hosts 映射


# ============================================================
# DNS和服务发现
# ============================================================

# Docker内置DNS工作原理:
# ├── 自动为每个容器创建DNS记录
# ├── 容器名 = DNS名
# ├── 容器名自动解析到容器IP
# └── 同一网络中的容器可以通过名称相互访问

# 测试DNS:
docker run --rm --network my-network alpine ping -c 1 container_name


# ============================================================
# 端口映射
# ============================================================

# 映射语法: -p 宿主机端口:容器端口

# 随机端口映射
docker run -d -p 8080 nginx  # 宿主机随机端口 -> 容器80

# 指定端口映射
docker run -d -p 8080:80 nginx  # 宿主机8080 -> 容器80

# 指定协议
docker run -d -p 8080:80/tcp nginx
docker run -d -p 8080:80/udp nginx

# 指定宿主机IP
docker run -d -p 127.0.0.1:8080:80 nginx  # 只有本地能访问

# 暴露所有端口
docker run -d -P nginx  # -P 将所有EXPOSE端口随机映射
```

## 5. Docker存储

### 5.1 存储类型详解

```
Docker存储 - 为什么要设计多种存储方式？

┌─────────────────────────────────────────────────────────────────────────┐
│                    数据持久化的挑战                                      │
│                                                                         │
│  问题: 容器内的数据在容器删除后会丢失!                                  │
│                                                                         │
│  例子:                                                                  │
│  ├── 数据库数据: 删除容器 = 删除数据库                                  │
│  ├── 用户上传文件: 删除容器 = 删除所有文件                             │
│  └── 日志数据: 容器重启后日志丢失                                       │
│                                                                         │
│  解决: 需要将数据存储在容器之外                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**Volumes (卷)**

```
Volumes - Docker管理的存储

原理:
┌────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  宿主机:                                                               │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  /var/lib/docker/volumes/                                       │ │
│  │  ┌──────────────────────────────────────────────────────────┐    │ │
│  │  │  my-volume/                                             │    │ │
│  │  │  ├── _data/                                              │    │ │
│  │  │  │   ├── file1.txt                                       │    │ │
│  │  │  │   └── file2.txt                                       │    │ │
│  │  │  └── metadata/                                           │    │ │
│  │  └──────────────────────────────────────────────────────────┘    │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                      │                                                │
│                      ▼                                                │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  容器内:                                                         │ │
│  │  /app/data/  <── 映射到宿主机 /var/lib/docker/volumes/...      │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘

为什么选择Volumes:
├── 容器无关: 不依赖宿主机特定路径
├── 隔离性: Docker管理，与宿主机文件系统隔离
├── 性能: 优化的存储驱动
├── 迁移: 可以轻松在不同宿主机间迁移
├── 备份: 可以使用Docker命令备份
└── 共享: 可以同时挂载到多个容器

使用场景:
├── 数据库数据存储
├── 应用日志存储
├── 需要持久化的配置文件
└── 容器间共享数据
```

**Bind Mounts (绑定挂载)**

```
Bind Mounts - 绑定宿主机路径

原理:
┌────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  宿主机:                                                               │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  /home/user/project/                                            │ │
│  │  ├── src/                                                       │ │
│  │  ├── config/                                                    │ │
│  │  └── .env                                                       │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│           │                                                          │
│           │ 直接映射                                                  │
│           ▼                                                          │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  容器内:                                                         │ │
│  │  /app/  <── 映射到宿主机 /home/user/project/                   │ │
│  │  ├── src/   (实时同步)                                         │ │
│  │  ├── config/ (实时同步)                                        │ │
│  │  └── .env   (实时同步)                                          │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘

为什么选择Bind Mounts:
├── 开发环境: 代码修改即时生效
├── 配置共享: 共享宿主机配置文件
├── 调试: 直接访问宿主机文件
└── 简单: 不需要Docker管理

使用场景:
├── 开发时代码热重载
├── 挂载宿主机配置文件
├── 访问宿主机日志
└── 调试用途

缺点:
├── 依赖宿主机路径
├── 安全性: 容器可以访问宿主机任意路径
└── 可移植性差
```

**tmpfs (临时文件系统)**

```
tmpfs - 内存存储

原理:
┌────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  宿主机内存:                                                           │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  ┌────────────────────────────────────────────────────────────┐  │ │
│  │  │  tmpfs数据存储在内存中                                     │  │ │
│  │  │  /sys/fs/cgroup/.../container-id/...                     │  │ │
│  │  └────────────────────────────────────────────────────────────┘  │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                      │                                                │
│                      ▼                                                │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  容器内:                                                         │ │
│  │  /tmp/  <── 存储在内存                                          │ │
│  │  /run/secrets/  <── 敏感数据                                   │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘

为什么选择tmpfs:
├── 性能: 内存访问比磁盘快得多
├── 安全: 容器删除后数据完全消失
└── 临时数据: 不需要持久化的数据

使用场景:
├── 缓存数据: 临时缓存
├── 敏感数据: 密码、密钥等
├── 性能敏感: 需要快速读写的临时数据
└── 会话数据: 用户会话信息

限制:
├── 内存大小限制
├── 容器停止后数据丢失
└── 占用内存，影响其他容器
```

### 5.2 存储操作详解

```bash
# ============================================================
# Volume 操作
# ============================================================

# 创建命名卷
docker volume create my-volume
#
# 参数:
# ├── --driver local: 存储驱动（默认）
# ├── --label: 标签
# └── --name: 卷名称

# 创建带配置的卷
docker volume create \
    --driver local \
    --opt type=none \
    --opt o=bind \
    --opt device=/path/on/host \
    my-bind-volume

# 列出所有卷
docker volume ls

# 查看卷详情
docker volume inspect my-volume
#
# 返回信息:
# {
#     "CreatedAt": "2024-01-01T00:00:00Z",
#     "Driver": "local",
#     "Labels": {},
#     "Mountpoint": "/var/lib/docker/volumes/my-volume/_data",
#     "Name": "my-volume",
#     "Options": null,
#     "Scope": "local"
# }

# 挂载卷到容器
docker run -d \
    --name web \
    -v my-volume:/app/data \           # 简写形式
    nginx

# 使用--mount方式（更明确）
docker run -d \
    --name web \
    --mount source=my-volume,target=/app/data,type=volume \
    nginx

# volume vs mount 区别:
# ├── -v my-volume:/app/data: 
# │   如果volume不存在会自动创建
# ├── --mount: 
# │   如果volume不存在会报错（更严格）
# └── 生产环境推荐使用 --mount

# 删除卷
docker volume rm my-volume

# 清理未使用的卷
docker volume prune

# 备份卷
docker run --rm \
    -v my-volume:/data \
    -v $(pwd):/backup \
    alpine \
    tar czf /backup/backup.tar.gz /data

# 恢复卷
docker run --rm \
    -v my-volume:/data \
    -v $(pwd):/backup \
    alpine \
    tar xzf /backup/backup.tar.gz -C /data


# ============================================================
# Bind Mounts 操作
# ============================================================

# 基本挂载
docker run -d \
    -v /host/path:/container/path \
    nginx

# 只读挂载
docker run -d \
    -v /host/path:/container/path:ro \
    nginx

# SELinux配置（如果遇到权限问题）
docker run -d \
    -v /host/path:/container/path:z \
    nginx  # 共享
docker run -d \
    -v /host/path:/container/path:Z \
    nginx  # 私有

# Windows路径注意:
docker run -d -v "C:\data:/app/data" nginx


# ============================================================
# tmpfs 操作
# ============================================================

# 使用tmpfs
docker run -d \
    --name web \
    --tmpfs /app/cache \
    nginx

# 详细tmpfs配置
docker run -d \
    --name web \
    --tmpfs /app/cache:size=100m,mode=1777 \
    nginx
#
# 参数:
# ├── size: 最大大小 (例如: 100m, 1g)
# └── mode: 权限 (例如: 1777)


# ============================================================
# 存储驱动选择
# ============================================================

# 查看当前存储驱动
docker info | grep "Storage Driver"

# 常见存储驱动:
# ├── overlay2: 推荐，用于生产环境
# ├── fuse-overlayfs: 用户空间overlay
# ├── vfs: 用于测试，简单但慢
# ├── btrfs: 需要宿主机是btrfs
# └── zfs: 需要宿主机是zfs

# 指定存储驱动启动
dockerd --storage-driver=overlay2
```

### 5.3 存储驱动原理

```
存储驱动 - 为什么要了解？

┌─────────────────────────────────────────────────────────────────────────┐
│                    存储驱动的作用                                        │
│                                                                         │
│  存储驱动负责:                                                          │
```

### 6.1 Docker Compose解决的问题

```

│                                                                         │

│  ├── 多个docker run命令繁琐                                            │

│  ├── 状态管理困难                                                       │
│  └── 无法一键启动整个应用                                               │

│                    Docker Compose的解决方案                              │

│  ├── 定义依赖关系                                                       │

│  ├── 可版本控制: 可以提交到Git                                          │
```


### 6.3 Compose配置详解

```yaml
# ============================================================
# version - 版本说明
# ============================================================
# 为什么要指定version:
# ├── 兼容性: 不同版本有不同的功能和语法
# ├── 推荐: 使用最新的稳定版本 (3.x 或 3.9)
# └── 注意: version在Compose V2中已废弃

version: '3.8'


# ============================================================
# services - 服务定义
# ============================================================
# 为什么要定义services:
# └── 每个服务对应一个容器

services:
  # 服务名称
  web:
    
    # build: 构建镜像
    # 为什么需要: 使用Dockerfile构建而非现成镜像
    build:
      context: .              # 构建上下文目录
      dockerfile: Dockerfile  # Dockerfile文件名
      args:                  # 构建参数
        NODE_ENV: production
      target: builder        # 多阶段构建的目标阶段
    
    # image: 指定镜像
    # 为什么需要: 使用现成镜像，不自己构建
    image: nginx:alpine
    
    # container_name: 容器名
    # 为什么需要: 明确指定容器名，便于管理
    container_name: my-web
    
    # restart: 重启策略
    # 为什么需要: 意外停止时自动重启
    restart: 
      # 选项:
      # ├── no: 不重启（默认）
      # ├── always: 总是重启
      # ├── on-failure: 失败时重启
      # └── unless-stopped: 除非手动停止
      - no
      - always
      - on-failure:3
      - unless-stopped
    
    # ports: 端口映射
    # 为什么需要: 让宿主机能访问容器服务
    ports:
      - "3000:3000"           # 宿主机:容器
      - "8080:80"
      - "443:443/tcp"
    
    # environment: 环境变量
    # 为什么需要: 配置应用运行环境
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgres://user:pass@db:5432/mydb
      - REDIS_URL=redis://redis:6379
    
    # env_file: 从文件加载环境变量
    # 为什么需要: 分离敏感配置
    env_file:
      - .env.common
      - .env.prod
    
    # volumes: 存储卷挂载
    # 为什么需要: 数据持久化
    volumes:
      - ./data:/app/data              # bind mount
      - volume-name:/var/lib/postgres # named volume
      - /app/node_modules             # 匿名卷（防止覆盖）
      - type: tmpfs                   # tmpfs
        target: /tmp
    
    # networks: 网络配置
    # 为什么需要: 服务间通信和隔离
    networks:
      - frontend
      - backend:
        aliases:
          - app.internal
    
    # depends_on: 依赖关系
    # 为什么需要: 控制启动顺序
    depends_on:
      db:
        condition: service_healthy  # 等待健康
      redis:
        condition: service_started # 等待启动
    
    # healthcheck: 健康检查
    # 为什么需要: 确保服务就绪
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    
    # deploy: 部署配置（Swarm模式）
    # 为什么需要: 生产环境部署配置
    deploy:
      replicas: 3                   # 副本数
      resources:                    # 资源限制
        limits:
          cpus: '1'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
      restart_policy:               # 重启策略
        condition: on-failure
        delay: 5s
        max_attempts: 3
      update_config:                # 更新配置
        parallelism: 2
        delay: 10s
        failure_action: rollback
    
    # command: 覆盖默认命令
    # 为什么需要: 使用不同方式启动
    command: npm start
    command: ["python", "manage.py", "migrate"]
    
    # entrypoint: 覆盖入口点
    entrypoint: /code/entrypoint.sh
    
    # user: 运行用户
    # 为什么需要: 安全考虑
    user: "1000:1000"
    
    # working_dir: 工作目录
    working_dir: /app
    
    # hostname: 主机名
    hostname: my-web
    
    # dns: DNS服务器
    dns: 8.8.8.8
    
    # extra_hosts: 添加hosts
    extra_hosts:
      - "host.docker.internal:host-gateway"


# ============================================================
# networks - 网络定义
# ============================================================
networks:
  frontend:
    driver: bridge          # 驱动类型
    ipam:                  # IP管理
      driver: default
      config:
        - subnet: 172.20.0.0/16
    external: false        # 是否外部网络
  
  backend:
    internal: true         # 内部网络（不能访问外网）
  
  # 外部网络（已存在的网络）
  external-network:
    external: true


# ============================================================
# volumes - 卷定义
# ============================================================
volumes:
  db-data:                 # 命名卷
    driver: local
  
  # 外部卷
  external-volume:
    external: true
  
  # 简短写法
  myvolume:
```
