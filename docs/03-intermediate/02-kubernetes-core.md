# Kubernetes核心概念

## 本章概述

Kubernetes（简称K8s）是容器编排的事实标准，已成为现代云原生应用的核心平台。本章将深入学习Kubernetes核心概念，不仅告诉你"怎么做"，更详细解释"为什么这样做"，帮助你理解每个概念背后的原理。

## 学习目标

- 深入理解Kubernetes架构原理和控制循环机制
- 掌握Pod生命周期管理和容器设计模式
- 深入理解各类Controller的工作机制和适用场景
- 全面掌握Service与Ingress的原理和配置
- 深入理解ConfigMap与Secret的区别和使用场景
- 理解持久化存储机制和存储类别

### 2.2 Pod结构详解

```yaml
# Pod完整配置 - 每个字段的原理
apiVersion: v1
kind: Pod
metadata:
  name: my-app           # Pod名称，集群内唯一
  namespace: default     # 所属命名空间，用于资源隔离
  labels:               # 标签，用于选择和管理
    app: my-app
    version: v1
  annotations:          # 注解，附加信息
    description: "My application pod"
spec:
  # ============================================================
  # containers - 容器定义
  # ============================================================
  containers:
  - name: app           # 容器名称（Pod内唯一）
    image: my-app:1.0   # 容器镜像
    
    # 镜像拉取策略 - 为什么需要？
    # ├── Always: 总是拉取最新镜像（默认）
    # ├── IfNotPresent: 本地有就用，没有就拉取
    # └── Never: 只用本地镜像
    imagePullPolicy: Always
    
    # 端口声明 - 为什么需要？
    # ├── 服务发现需要
    # ├── 文档作用
    # └── 健康检查需要
    ports:
    - containerPort: 8080
      protocol: TCP
    
    # 资源限制 - 为什么需要？
    # ├── 防止容器耗尽资源
    # ├── 调度依据
    # └── 计费依据
    resources:
      requests:          # 调度时的资源需求
        memory: "128Mi"  # 128MB内存
        cpu: "100m"      # 0.1 CPU核心
      limits:            # 资源上限
        memory: "256Mi"
        cpu: "500m"
    
    # 环境变量 - 为什么需要？
    # ├── 配置应用
    # ├── 敏感信息（不推荐，用Secret）
    # └── 动态配置
    env:
    - name: ENV
      value: "production"
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database_host
    
    # 卷挂载 - 为什么需要？
    # ├── 持久化数据
    # ├── 共享配置
    # └── 临时存储
    volumeMounts:
    - name: data
      mountPath: /data
    
    # 健康检查 - 为什么需要？
    # ├── livenessProbe: 容器是否存活，存活则不重启
    # ├── readinessProbe: 容器是否准备好接收流量
    # └── startupProbe: 容器是否启动完成
    livenessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 30  # 启动后多久开始检查
      periodSeconds: 10       # 检查间隔
      timeoutSeconds: 5       # 检查超时
      failureThreshold: 3      # 连续失败次数才重启
    
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
    
    # 启动和停止钩子 - 为什么需要？
    # ├── postStart: 容器启动后执行
    # └── preStop: 容器停止前执行（如优雅关闭）
    lifecycle:
      postStart:
        exec:
          command: ["/bin/sh", "-c", "echo 'Started'"]
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 10"]

  # 初始化容器 - 为什么需要？
  # ├── 主容器运行前完成初始化
  # ├── 准备依赖环境
  # └── 只运行一次
  initContainers:
  - name: init
    image: busybox
    command: ['sh', '-c', 'echo "Initializing..."']

  # 卷定义 - 为什么需要？
  # ├── 数据持久化
  # └── 容器间共享
  volumes:
  - name: data
    emptyDir: {}  # 临时存储，Pod删除后丢失

  # 重启策略 - 为什么需要？
  # ├── Always: 总是重启（默认）
  # ├── OnFailure: 失败时重启
  # └── Never: 不重启
  restartPolicy: Always

  # 终止宽限期 - 为什么需要？
  # 优雅终止的时间，给应用时间处理SIGTERM
  terminationGracePeriodSeconds: 30
```

阶段详解:

Pending (等待中)
├── 含义: Pod已创建，等待调度
├── 可能原因:
│   ├── 等待Scheduler分配节点
│   ├── 拉取镜像中
│   └── 等待依赖资源
└── 排查: 查看Events

Running (运行中)
├── 含义: Pod已调度到节点，至少一个容器运行
├── 容器状态:
│   ├── Running: 正常运行
│   ├── Waiting: 等待中
│   └── Terminating: 正在终止

├── 含义: 所有容器正常退出，不会重启
└── 适用: Job、一次性任务

Failed (失败)
├── 含义: 至少一个容器异常退出
├── 可能原因:
│   ├── 应用程序错误
│   ├── 健康检查失败
│   └── 资源限制超出
└── 排查: 查看容器日志

Unknown (未知)
├── 含义: 无法获取Pod状态
├── 可能原因:
│   ├── 节点通信故障
│   └── kubelet异常
└── 排查: 检查节点状态

### 3.2 Deployment - 无状态应用管理

```yaml
# Deployment完整配置

kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  # ============================================================
  # replicas - 副本数
  # ============================================================
  # 为什么需要:
  # ├── 高可用: 多个副本确保服务持续可用
  # ├── 负载均衡: 流量分散到多个实例
  # └── 容错: 单个故障不影响整体服务
  replicas: 3

  # ============================================================
  # selector - 标签选择器
  # ============================================================
  # 为什么需要:
  # └── Controller需要知道管理哪些Pod
  selector:
    matchLabels:
      app: web

  # ============================================================
  # strategy - 更新策略
  # ============================================================
  # 为什么需要:
  # └── 控制更新过程中的行为
  strategy:
    type: RollingUpdate    # 滚动更新
    # 或 Recreate: 删除所有再创建
    
    rollingUpdate:
      # maxSurge: 额外可以启动多少个Pod
      # ├── 数字: 如1
      # ├── 百分比: 如25%
      # └── 含义: 允许超过期望副本数的最大数量
      maxSurge: 1
      
      # maxUnavailable: 可以不可用多少个Pod
      # ├── 数字: 如0
      # ├── 百分比: 如25%
      # └── 含义: 更新过程中保持可用的最小副本数
      maxUnavailable: 0

  # ============================================================
  # template - Pod模板
  # ============================================================
  # 为什么需要:
  # └── 定义如何创建Pod
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "50m"
          limits:
            memory: "128Mi"
            cpu: "100m"

## 6. 持久化存储

### 6.1 为什么需要持久化存储？

```

### 6.2 PV/PVC架构

```
为什么需要两层抽象 (PV + PVC)？

┌─────────────────────────────────────────────────────────────────────────┐
│                    架构设计理念                                          │
│                                                                         │
│  PV (PersistentVolume):                                                 │
│  ├── 集群级资源: 管理员创建                                            │
│  ├── 实际存储: 对接各种存储后端                                         │
│  ├── 与Pod解耦: PV独立于Pod生命周期                                    │
│  └── 静态/动态: 可以预创建或按需创建                                   │
│                                                                         │
│  PVC (PersistentVolumeClaim):                                          │
│  ├── 命名空间级: 用户创建                                              │
│  ├── 声明需求: 声明存储大小、访问模式                                   │
│  ├── 绑定: 自动匹配符合要求的PV                                        │
│  └── 消费: Pod使用PVC挂载                                              │
│                                                                         │
│  好处:                                                                  │
│  ├── 存储与计算分离                                                   │
│  ├── 开发者不需要了解存储细节                                           │
│  ├── 支持多种存储类型                                                  │
│  └── 便于存储管理                                                      │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│                    PV/PVC/Pod 关系                                     │
│                                                                         │
│   ┌──────────────┐         ┌──────────────┐         ┌──────────────┐ │
│   │      PV      │◄────────│     PVC     │◄────────│     Pod      │ │
│   │              │  绑定    │              │  挂载    │              │ │
│   │  集群级资源  │         │  命名空间级  │         │ 使用存储     │ │
│   │  管理员管理  │         │  用户管理    │         │              │ │
│   └──────────────┘         └──────────────┘         └──────────────┘ │
│                                                                         │
│   流程:                                                                 │
│   1. 管理员创建PV (如10Gi SSD)                                        │
│   2. 用户创建PVC (请求5Gi)                                             │
│   3. Kubernetes自动绑定                                               │
│   4. Pod使用PVC挂载存储                                               │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.3 PV配置详解

```yaml
# PersistentVolume - 多种存储类型
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-data
spec:
  # ============================================================
  # capacity - 容量
  # ============================================================
  capacity:
    storage: 10Gi

  # ============================================================
  # accessModes - 访问模式
  # ============================================================
  # 为什么需要:
  # └── 定义存储的访问方式
  # ├── ReadWriteOnce (RWO): 单节点读写
  # ├── ReadOnlyMany (ROX): 多节点只读
  # └── ReadWriteMany (RWX): 多节点读写
  accessModes:
  - ReadWriteOnce

  # ============================================================
  # persistentVolumeReclaimPolicy - 回收策略
  # ============================================================
  # 为什么需要:
  # └── PVC删除后的处理方式
  # ├── Retain: 保留数据（手动处理）
  # ├── Delete: 删除存储（云存储常用）
  # └── Recycle: 删除数据后重用（已废弃）
  persistentVolumeReclaimPolicy: Retain

  # ============================================================
  # storageClassName - 存储类
  # ============================================================
  # 为什么需要:
  # └── 动态创建PV时指定存储类型
  storageClassName: standard

  # ============================================================
  # 方式1: HostPath (本地存储)
  # ============================================================
  hostPath:
    path: /mnt/data
    type: DirectoryOrCreate

  # 方式2: NFS
  # nfs:
  #   server: nfs-server.example.com
  #   path: /exports

  # 方式3: 云存储
  # awsElasticBlockStore:
  #   volumeID: vol-12345
  #   fsType: ext4

  # 方式4: GCE Persistent Disk
  # gcePersistentDisk:
  #   pdName: my-disk
  #   fsType: ext4
```

### 6.4 PVC配置详解

```yaml
# PersistentVolumeClaim

  name: pvc-data

  # 访问模式
  accessModes:
  - ReadWriteOnce

  # 存储类（空表示不使用动态创建）
  storageClassName: standard

  # 资源请求
  resources:
    requests:
      storage: 10Gi

# Pod中使用PVC
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: pvc-data
```

### 6.5 StorageClass详解

```
为什么需要StorageClass？

┌─────────────────────────────────────────────────────────────────────────┐
│                    静态 vs 动态 Provisioning                            │
│                                                                         │
│  静态 (Static):                                                        │
│  ├── 管理员手动创建PV                                                  │
│  ├── 用户创建PVC绑定到现有PV                                           │
│  └── 问题: 需要手动管理大量PV                                          │
│                                                                         │
│  动态 (Dynamic):                                                       │
│  ├── 使用StorageClass自动创建PV                                        │
│  ├── 用户只需创建PVC                                                   │
│  ├── 存储系统自动创建实际存储                                           │
│  └── 优势: 无需预创建，按需使用                                        │
└─────────────────────────────────────────────────────────────────────────┘

常用StorageClass:

┌─────────────────────────────────────────────────────────────────────────┐
│     Provider    │   类型      │                    说明                 │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ standard       │   HDD       │ 默认存储，适合一般工作负载              │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ pd-ssd         │   SSD       │ 高性能，适合数据库                     │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ pd-standard    │   HDD       │ GCP默认                                 │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ gp2/gp3       │   SSD       │ AWS通用SSD                              │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ io1/io2       │   NVMe      │ AWS高性能SSD                           │
├────────────────┼─────────────┼─────────────────────────────────────────┤
│ managed-premium│   SSD       │ Azure高级SSD                           │
└────────────────┴─────────────┴─────────────────────────────────────────┘
```
