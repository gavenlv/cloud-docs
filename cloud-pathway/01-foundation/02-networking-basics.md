## 1. OSI七层模型

### 1.1 OSI模型概述

OSI（Open Systems Interconnection）模型是网络通信的理论框架，将网络通信分为七个层次：

```
┌─────────────────────────────────────────────────────────────┐
│                    OSI七层模型                               │
├─────────────┬───────────────┬───────────────────────────────┤
│    层级     │     名称      │           功能说明             │
├─────────────┼───────────────┼───────────────────────────────┤
│    第7层    │    应用层     │ 用户接口，HTTP/FTP/SMTP        │
│    第6层    │    表示层     │ 数据格式转换，加密/解密        │
│    第5层    │    会话层     │ 建立和维护会话                 │
│    第4层    │    传输层     │ 端到端传输，TCP/UDP            │
│    第3层    │    网络层     │ 路由选择，IP协议               │
│    第2层    │  数据链路层   │ 物理寻址，以太网帧             │
│    第1层    │    物理层     │ 比特流传输，网线/光纤          │
└─────────────┴───────────────┴───────────────────────────────┘
```

### 1.2 数据封装过程

```
发送端数据封装过程：

应用层    ┌─────────────┐
          │    DATA     │  原始数据
          └──────┬──────┘
传输层    ┌──────┴──────┐
          │ TCP │ DATA  │  添加TCP头
          └──────┬──────┘
网络层    ┌──────┴──────┐
          │ IP  │TCP│DATA│  添加IP头
          └──────┬──────┘
数据链路层 ┌──────┴──────────┐
          │ ETH │IP │TCP│DATA│  添加以太网帧头
          └──────┬──────────┘
物理层          比特流传输
```

### 1.3 各层常见协议

| 层级 | 常见协议 |
|-----|---------|
| 应用层 | HTTP, HTTPS, FTP, SSH, DNS, SMTP, POP3, IMAP |
| 传输层 | TCP, UDP |
| 网络层 | IP, ICMP, ARP, OSPF, BGP |
| 数据链路层 | Ethernet, PPP, WiFi |
| 物理层 | RJ45, 光纤, 无线电波 |

## 2. TCP/IP协议栈

### 2.1 TCP/IP模型与OSI对比

```
OSI模型                    TCP/IP模型
┌─────────────┐           ┌─────────────┐
│    应用层    │           │             │
├─────────────┤           │    应用层    │
│    表示层    │           │             │
├─────────────┤           │             │
│    会话层    │           │             │
├─────────────┤           ├─────────────┤
│    传输层    │           │    传输层    │
├─────────────┤           ├─────────────┤
│    网络层    │           │    网络层    │
├─────────────┤           ├─────────────┤
│  数据链路层  │           │             │
├─────────────┤           │  网络接口层  │
│    物理层    │           │             │
└─────────────┘           └─────────────┘
```

### 2.2 TCP协议详解

**TCP（传输控制协议）特点**：
- 面向连接
- 可靠传输
- 流量控制
- 拥塞控制

**TCP三次握手**：
```
客户端                              服务端
   │                                  │
   │─────── SYN ─────────────────────►│  第1次：请求建立连接
   │                                  │
   │◄────── SYN + ACK ───────────────│  第2次：确认并同意连接
   │                                  │
   │─────── ACK ─────────────────────►│  第3次：确认收到
   │                                  │
   │         连接建立完成              │
```

**TCP四次挥手**：
```
客户端                              服务端
   │                                  │
   │─────── FIN ─────────────────────►│  第1次：请求关闭连接
   │                                  │
   │◄────── ACK ─────────────────────│  第2次：确认收到
   │                                  │
   │◄────── FIN ─────────────────────│  第3次：服务端请求关闭
   │                                  │
   │─────── ACK ─────────────────────►│  第4次：确认关闭
   │                                  │
   │         连接关闭完成              │
```

### 2.3 UDP协议详解

**UDP（用户数据报协议）特点**：
- 无连接
- 不可靠传输
- 开销小、速度快
- 不保证顺序

**TCP vs UDP对比**：

| 特性 | TCP | UDP |
|-----|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 顺序 | 有序 | 无序 |
| 速度 | 较慢 | 快 |
| 开销 | 大 | 小 |
| 应用场景 | Web、邮件、文件传输 | 视频、音频、游戏 |

## 3. IP地址与子网

### 3.1 IP地址基础

**IPv4地址**：32位，通常表示为4个十进制数（点分十进制）

```
IP地址示例：192.168.1.100

二进制表示：
11000000.10101000.00000001.01100100

每段范围：0-255
```

**IP地址分类**：

| 类别 | 范围 | 默认子网掩码 | 私有地址范围 |
|-----|------|-------------|-------------|
| A类 | 1.0.0.0 - 126.255.255.255 | 255.0.0.0 | 10.0.0.0 - 10.255.255.255 |
| B类 | 128.0.0.0 - 191.255.255.255 | 255.255.0.0 | 172.16.0.0 - 172.31.255.255 |
| C类 | 192.0.0.0 - 223.255.255.255 | 255.255.255.0 | 192.168.0.0 - 192.168.255.255 |

### 3.2 子网掩码

**子网掩码作用**：区分网络部分和主机部分

```
IP地址：    192.168.1.100
子网掩码：  255.255.255.0
            ────────────
网络部分：  192.168.1.0
主机部分：  0.0.0.100

计算过程：
IP地址：    11000000.10101000.00000001.01100100
子网掩码：  11111111.11111111.11111111.00000000
网络地址：  11000000.10101000.00000001.00000000 (192.168.1.0)
广播地址：  11000000.10101000.00000001.11111111 (192.168.1.255)
```

### 3.3 CIDR表示法

**CIDR（无类别域间路由）**：用斜线表示网络前缀长度

```
CIDR表示：192.168.1.0/24

/24 表示前24位是网络部分

常见CIDR：
/8   = 255.0.0.0       约1677万台主机
/16  = 255.255.0.0     约65534台主机
/24  = 255.255.255.0   254台主机
/27  = 255.255.255.224 30台主机
/30  = 255.255.255.252 2台主机（常用于点对点链路）
```

### 3.4 子网划分计算

**示例**：将 192.168.1.0/24 划分为4个子网

```
原网络：192.168.1.0/24
需要子网数：4
需要借用主机位：2位（2^2 = 4）

新子网掩码：/24 + 2 = /26

子网划分结果：
┌──────────────┬─────────────────┬─────────────────┬─────────────┐
│   子网       │    网络地址     │    可用主机范围   │  广播地址   │
├──────────────┼─────────────────┼─────────────────┼─────────────┤
│ 子网1        │ 192.168.1.0/26  │ .1 - .62        │ .63         │
│ 子网2        │ 192.168.1.64/26 │ .65 - .126      │ .127        │
│ 子网3        │ 192.168.1.128/26│ .129 - .190     │ .191        │
│ 子网4        │ 192.168.1.192/26│ .193 - .254     │ .255        │
└──────────────┴─────────────────┴─────────────────┴─────────────┘
```

**计算公式**：
- 子网数 = 2^借位数
- 每个子网主机数 = 2^(主机位数-借位数) - 2
- 可用主机数 = 总主机数 - 2（网络地址和广播地址）

## 4. DNS域名系统

### 4.1 DNS作用

DNS（Domain Name System）将域名解析为IP地址，方便用户记忆和访问。

```
用户输入：www.example.com
DNS解析：→ 93.184.216.34
```

### 4.2 DNS层级结构

```
DNS层级结构（从右到左）：

                    ┌─────────┐
                    │   根域   │  (.)
                    └────┬────┘
                         │
        ┌────────────────┼────────────────┐
        │                │                │
   ┌────┴────┐      ┌────┴────┐      ┌────┴────┐
   │   .com  │      │   .org  │      │   .cn   │  顶级域(TLD)
   └────┬────┘      └─────────┘      └─────────┘
        │
   ┌────┴────┐
   │ example │                          二级域
   └────┬────┘
        │
   ┌────┴────┐
   │   www   │                          主机名
   └─────────┘
```

### 4.3 DNS解析过程

```
                    DNS解析流程
                    
用户电脑                本地DNS               权威DNS
   │                      │                     │
   │───查询www.example.com─►│                     │
   │                      │                     │
   │                      │───查询根服务器───────►│
   │                      │◄──返回.com服务器地址──│
   │                      │                     │
   │                      │───查询.com服务器─────►│
   │                      │◄──返回example.com地址─│
   │                      │                     │
   │                      │───查询权威DNS────────►│
   │                      │◄──返回IP地址─────────│
   │                      │                     │
   │◄──返回IP地址──────────│                     │
   │                      │                     │
```

### 4.4 DNS记录类型

| 记录类型 | 说明 | 示例 |
|---------|------|------|
| A | 域名到IPv4地址 | www.example.com → 192.168.1.1 |
| AAAA | 域名到IPv6地址 | www.example.com → 2001:db8::1 |
| CNAME | 别名记录 | www.example.com → example.com |
| MX | 邮件服务器 | example.com → mail.example.com |
| NS | 域名服务器 | example.com → ns1.example.com |
| TXT | 文本记录 | 用于SPF、验证等 |
| SOA | 起始授权记录 | 区域授权信息 |

## 5. HTTP/HTTPS协议

### 5.1 HTTP协议基础

**HTTP（超文本传输协议）**：应用层协议，用于Web通信

**HTTP请求结构**：
```
GET /index.html HTTP/1.1          请求行
Host: www.example.com             请求头
User-Agent: Mozilla/5.0
Accept: text/html
Content-Type: application/json
                                  空行
{"key": "value"}                  请求体（POST请求）
```

**HTTP响应结构**：
```
HTTP/1.1 200 OK                   状态行
Content-Type: text/html           响应头
Content-Length: 1234
Server: nginx
                                  空行
<!DOCTYPE html>...                响应体
```

### 5.2 HTTP方法

| 方法 | 说明 | 幂等性 |
|-----|------|--------|
| GET | 获取资源 | 是 |
| POST | 创建资源 | 否 |
| PUT | 更新资源（完整替换） | 是 |
| PATCH | 部分更新 | 否 |
| DELETE | 删除资源 | 是 |
| HEAD | 获取响应头 | 是 |
| OPTIONS | 获取支持的方法 | 是 |

### 5.3 HTTP状态码

```
状态码分类：
┌──────────┬─────────────────────────────────────────┐

├──────────┼─────────────────────────────────────────┤
│  2xx     │  成功 - 请求已成功接收、理解、接受        │
│  200 OK  │  请求成功                               │
│  201 Created │ 创建成功                            │
├──────────┼─────────────────────────────────────────┤
│  3xx     │  重定向 - 需要进一步操作                 │

├──────────┼─────────────────────────────────────────┤
│  4xx     │  客户端错误 - 请求包含语法错误           │
│  400 Bad Request │ 请求格式错误                   │
```

## 6. 负载均衡基础

### 6.1 负载均衡概念

**负载均衡**：将流量分发到多个服务器，提高可用性和性能

```
                    负载均衡器
                        │
        ┌───────────────┼───────────────┐
        │               │               │
        ▼               ▼               ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Server1 │    │ Server2 │    │ Server3 │
   └─────────┘    └─────────┘    └─────────┘
```

### 6.2 负载均衡算法

| 算法 | 说明 | 适用场景 |
|-----|------|---------|
| 轮询(Round Robin) | 依次分发 | 服务器性能相近 |
| 加权轮询 | 按权重分发 | 服务器性能不同 |
| 最少连接 | 分发到连接最少的服务器 | 长连接场景 |
| IP哈希 | 根据IP哈希分发 | 需要会话保持 |
| URL哈希 | 根据URL哈希分发 | 缓存场景 |

### 6.3 四层与七层负载均衡

```
四层负载均衡（L4）              七层负载均衡（L7）
┌─────────────────┐          ┌─────────────────┐
│ 基于IP+端口分发  │          │ 基于URL/Host分发 │
│ 性能高          │          │ 功能丰富         │
│ 不检查内容      │          │ 可做内容路由     │
│ TCP/UDP级别     │          │ HTTP级别         │
└─────────────────┘          └─────────────────┘
     LVS、F5                    Nginx、HAProxy
```

## 6.4 网络协议底层原理深度解析

### 6.4.1 TCP三次握手深度原理

**TCP为什么需要三次握手？**

```
┌─────────────────────────────────────────────────────────────────┐
│              TCP三次握手的必要性                                    │
└─────────────────────────────────────────────────────────────────┘

核心问题：如何在不可靠的网络中建立可靠连接？

┌─────────────────────────────────────────────────────────────────┐
│  两次握手的问题：                                                 │
│                                                                  │
│  场景：Client发送SYN但网络延迟                                    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  时间轴：                                                │   │
│  │                                                          │   │
│  │  T1: Client → SYN → Server (旧连接请求)               │   │
│  │  T2: Client → SYN → Server (新连接请求)               │   │
│  │  T3: Server → SYN-ACK → Client                        │   │
│  │  T4: 连接建立！                                        │   │
│  │  T5: 旧SYN到达Server                                   │   │
│  │  T6: Server → SYN-ACK → Client                        │   │
│  │  T7: 再次建立连接！(错误)                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  结果：Server维护了两个连接，资源浪费                           │
└─────────────────────────────────────────────────────────────────┘

三次握手解决什么问题：

┌─────────────────────────────────────────────────────────────────┐
│  握手过程：                                                       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第一次：Client → SYN(seq=x) → Server                   │   │
│  │  └── Client发送初始序列号x                              │   │
│  │                                                          │   │
│  │  第二次：Server → SYN-ACK(seq=y, ack=x+1) → Client     │   │
│  │  └── Server发送初始序列号y                              │   │
│  │  └── 确认收到x                                          │   │
│  │                                                          │   │
│  │  第三次：Client → ACK(seq=x+1, ack=y+1) → Server      │   │
│  │  └── 确认收到y                                          │   │
│  │                                                          │   │
│  │  连接建立！                                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  序列号作用：                                                     │
│  - 每个字节都有一个唯一的序列号                                  │
│  - 用于数据排序和丢包检测                                        │
│  - 防止旧数据包被误认为是新数据                                 │
└─────────────────────────────────────────────────────────────────┘

TCP状态转换图：

┌─────────────────────────────────────────────────────────────────┐
│  客户端状态变化：                                                 │
│                                                                  │
│  CLOSED → SYN-SENT → ESTABLISHED → FIN-WAIT-1                 │
│                    ↓                    ↓                       │
│              可能超时重传          发送数据/接收ACK              │
│                                                                  │
│  服务器端状态变化：                                               │
│                                                                  │
│  CLOSED → LISTEN → SYN-RCVD → ESTABLISHED                     │
│                      ↓              ↓                           │
│              接收SYN          接收ACK                           │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4.2 TCP四次挥手深度原理

**TCP为什么需要四次挥手？**

```
┌─────────────────────────────────────────────────────────────────┐
│              TCP四次挥手的必要性                                    │
└─────────────────────────────────────────────────────────────────┘

为什么不能像三次握手一样合并？

┌─────────────────────────────────────────────────────────────────┐
│  原因：TCP是全双工通信                                           │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  全双工意味着：                                          │   │
│  │  - 双方可以同时发送数据                                 │   │
│  │  - 发送通道和接收通道是独立的                           │   │
│  │  - 关闭时需要分别关闭两个通道                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  挥手过程：                                                       │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Client → FIN → Server  (Client不再发送数据)           │   │
│  │  Server → ACK → Client  (确认收到FIN)                 │   │
│  │                                                          │   │
│  │  此时：Client不能发送数据，但可以接收数据                │   │
│  │                                                          │   │
│  │  Server继续处理数据...                                   │   │
│  │                                                          │   │
│  │  Server → FIN → Client  (Server不再发送数据)           │   │
│  │  Client → ACK → Server  (确认收到FIN)                 │   │
│  │                                                          │   │
│  │  连接关闭！                                              │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

TIME_WAIT状态的作用：

┌─────────────────────────────────────────────────────────────────┐
│  为什么Client在TIME_WAIT等待2MSL？                              │
│                                                                  │
│  MSL (Maximum Segment Lifetime):                                │
│  - IP数据包在网络中最大存活时间                                  │
│  - 典型值：60秒、120秒                                          │
│                                                                  │
│  目的：                                                          │
│  1. 确保Server收到最后的ACK                                      │
│     - 如果ACK丢失，Server会重发FIN                             │
│     - Client在TIME_WAIT可以重发ACK                              │
│                                                                  │
│  2. 确保旧连接的数据包消失                                       │
│     - 防止延迟的数据包被新连接误认为                            │
│     - 避免数据混淆                                              │
│                                                                  │
│  2MSL = 往返最大时间                                             │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4.3 DNS递归查询深度原理

**DNS查询是怎么工作的？**

```
┌─────────────────────────────────────────────────────────────────┐
│              DNS查询流程深度解析                                   │
└─────────────────────────────────────────────────────────────────┘

DNS层级结构：

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   . (根域名服务器)                                              │
│      │                                                         │
│      ├── .com (顶级域名服务器)                                  │
│      │      │                                                  │
│      │      ├── google.com (权威域名服务器)                   │
│      │      │                                                 │
│      │      └── example.com (权威域名服务器)                  │
│      │                                                        │
│      ├── .org (顶级域名服务器)                                  │
│      │                                                        │
│      └── .cn (顶级域名服务器)                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

递归查询 vs 迭代查询：

┌─────────────────────────────────────────────────────────────────┐
│  递归查询（Resolver完成全部查询）：                               │
│                                                                  │
│  Client → DNS Resolver → Root Server                           │
│                        → .com Server                            │
│                        → example.com Server                    │
│                        → 返回结果给Client                       │
│                                                                  │
│  特点：Resolver代替Client完成全部查询                            │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  迭代查询（各服务器只返回下一跳）：                      │   │
│  │                                                          │   │
│  │  Client → DNS Resolver → Root Server                    │   │
│  │         (返回.com服务器地址)                             │   │
│  │           → .com Server                                 │   │
│  │         (返回example.com服务器地址)                     │   │
│  │           → example.com Server                         │   │
│  │         (返回真实IP)                                     │   │
│  │           → 返回结果给Resolver → Client                 │   │
│  │                                                          │   │
│  │  特点：各服务器只返回提示，不代替查询                    │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

DNS缓存机制：

┌─────────────────────────────────────────────────────────────────┐
│  DNS查询结果缓存：                                               │
│                                                                  │
│  缓存位置：                                                      │
│  1. 浏览器DNS缓存                                               │
│  2. 操作系统DNS缓存 (hosts文件)                                │
│  3. 路由器DNS缓存                                               │
│  4. ISP DNS服务器缓存                                           │
│  5. 递归Resolver缓存                                            │
│                                                                  │
│  缓存时间由TTL决定：                                             │
│  - A记录：通常300秒到86400秒                                    │
│  - NS记录：通常86400秒到172800秒                                │
│  - CNAME：通常300秒到3600秒                                     │
│                                                                  │
│  缓存问题：                                                      │
│  - DNS变更后需要等待TTL过期才能生效                              │
│  - 缓存投毒攻击风险                                              │
│  - DNSSEC可以解决缓存安全问题                                   │
└─────────────────────────────────────────────────────────────────┘
```
